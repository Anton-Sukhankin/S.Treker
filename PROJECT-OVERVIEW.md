# S-Tracker: как устроен проект изнутри

Этот документ объясняет проект простым языком. Он написан не для разработчика, а для человека, который хочет понять: что в проекте есть, как одно связано с другим и почему интерфейс работает именно так.

Главная мысль: **S-Tracker - это прототип списка задач, где задачи можно смотреть по направлениям работы, очередям, статусам, фильтрам и колонкам**.

Можно представить проект как большой шкаф с документами:

- в шкафу лежат задачи;
- задачи разложены по рабочим направлениям;
- у каждой задачи есть очередь в рабочем маршруте и собственный статус;
- у каждой задачи есть набор признаков;
- пользователь выбирает, какую часть шкафа открыть и какие признаки показать на столе.

## 1. Что является главным в проекте

Главная сущность проекта - **задача**.

Задача - это одна строка в таблице. Например:

- согласовать договор;
- проверить закупку;
- обработать поручение;
- проконтролировать замечание.

У задачи есть основные данные:

- номер;
- название;
- направление работы;
- статус;
- исполнитель;
- автор;
- дата создания;
- срок;
- дополнительные признаки.

Дополнительные признаки в проекте называются **атрибутами**.

Проще: атрибут - это любое поле задачи. Например:

- тип документа;
- контрагент;
- сумма;
- объект;
- критичность;
- SLA;
- поставщик;
- дата поставки.

У задачи также есть настоящий **статус** как отдельный атрибут. Он показывает состояние самой задачи: например, "Новая", "Назначена", "В работе", "Завершена", "Отклонена".

Важно не путать:

- **очередь** - где задача находится в рабочем маршруте и почему она попала в верхнюю вкладку;
- **статус** - состояние самой задачи, которое показывается цветным тегом в таблице и карточке.

## 2. Как устроен основной процесс работы с задачами

Основной процесс выглядит так:

1. Пользователь выбирает, какие задачи он хочет смотреть.
2. Потом выбирает очередь задач.
3. Потом при необходимости использует поиск.
4. Потом может применить фильтры.
5. Потом может настроить колонки таблицы.

То есть порядок примерно такой:

**бизнес-домен -> очередь -> поиск и фильтры -> таблица -> настройка колонок**

Разберем это по шагам.

## 3. Что такое бизнес-домен

**Бизнес-домен** - это рабочее направление.

В текущем прототипе показаны три бизнес-домена:

- строительные договоры;
- закупки ТМЦ;
- поручения СД.

Это не полный и не постоянный список для всей системы. Эти три домена относятся к текущему демонстрационному набору, фактически к одной роли пользователя.

В будущей системе у разных ролей может быть разный набор бизнес-доменов. У одной роли их может быть больше, у другой меньше, у третьей - совсем другой состав.

Можно представить бизнес-домен как отдельную папку с задачами.

Например:

- в папке "Строительные договоры" лежат задачи по договорам;
- в папке "Закупки ТМЦ" лежат задачи по закупкам;
- в папке "Поручения СД" лежат задачи по поручениям и контролю.

В левой панели пользователь выбирает одну из таких папок.

Еще есть пункт **"Все задачи"**. Это не отдельный домен, а общий режим. Он показывает задачи сразу из всех доменов, доступных текущей роли пользователя.

## 4. Что такое очередь

Над таблицей есть очереди:

- к исполнению;
- в работе;
- обработаны мной;
- к назначению;
- наблюдаю;
- инициированы мной.

Внутри проекта эти значения нужно понимать как **очереди задач**, а не как настоящий статус задачи.

Очередь отвечает на вопрос: "Где сейчас задача находится относительно пользователя и рабочего маршрута?"

Например:

- "К исполнению" - задачи, которые нужно взять и сделать;
- "В работе" - задачи, которые уже выполняются;
- "К назначению" - задачи, которым нужно назначить исполнителя;
- "Наблюдаю" - задачи, за которыми пользователь следит;
- "Инициированы мной" - задачи, которые пользователь сам запустил.

Важно: бизнес-домен и очередь работают вместе.

Например:

- пользователь выбрал "Закупки ТМЦ";
- потом выбрал очередь "В работе";
- значит таблица показывает только задачи по закупкам, которые сейчас в работе.

Если пользователь выбрал "Все задачи" и очередь "К исполнению", таблица показывает задачи к исполнению по всем доменам, доступным его текущей роли.

## 5. Как связаны бизнес-домены, очереди, фильтры и таблица

Связь такая:

1. **Бизнес-домен** определяет, из какой рабочей области брать задачи.
2. **Очередь** определяет, какие задачи из этой области показать.
3. **Фильтры** дополнительно сужают список.
4. **Таблица** показывает итоговый результат.
5. **Колонки** определяют, какие признаки задач видны в таблице.

Простой пример:

Пользователь выбирает:

- домен: "Строительные договоры";
- очередь: "К исполнению";
- фильтр: контрагент "ООО СтройПроект";
- колонки: номер, название, тип документа, контрагент, сумма.

В итоге таблица показывает:

- только строительные договоры;
- только задачи к исполнению;
- только задачи с выбранным контрагентом;
- только выбранные колонки.

То есть таблица не живет сама по себе. Она всегда является результатом нескольких выборов пользователя.

## 6. Откуда берутся данные

Сейчас данные не приходят с сервера. Они лежат внутри проекта.

Главное место с данными:

`src/data/task-model.js`

Отдельный справочник людей:

`src/data/users.js`

В этом файле описаны:

- бизнес-домены;
- текущая роль пользователя;
- какие домены доступны роли;
- очереди задач;
- статусы задач;
- список всех атрибутов;
- какие атрибуты относятся к каким доменам;
- какие атрибуты видны сразу;
- какие атрибуты лежат в библиотеке;
- справочники значений;
- сами задачи;
- значения атрибутов задач.

В `src/data/users.js` лежат люди, которые могут быть авторами, исполнителями или наблюдателями задач. Задача хранит не все данные автора, а только ссылку на человека, например `createdBy`. Потом система по этой ссылке находит нужного человека и показывает в интерфейсе его аватар, фамилию, имя, отчество и email.

Проще говоря, этот файл отвечает на вопрос:

**"Что вообще есть в системе?"**

Например, там хранится знание, что:

- есть домен "Закупки ТМЦ";
- у него есть атрибуты "Бизнес-юнит", "SLA", "Срок", "Поставщик";
- часть этих атрибутов видна в таблице сразу;
- часть доступна только через библиотеку атрибутов;
- задачи этого домена имеют значения по этим атрибутам.

## 7. Где лежит логика, которая собирает данные для интерфейса

Данные сами по себе еще не умеют отображаться.

Чтобы интерфейс понял, что показать пользователю, есть отдельный слой правил:

`src/domain`

Эту папку можно представить как "переводчика" между сырыми данными и экраном.

Например:

- данные знают, какие домены существуют;
- правила определяют, какие из них доступны текущей роли;
- данные знают, что есть много задач;
- правила выбирают только задачи нужного домена и нужной очереди;
- данные знают, что есть много атрибутов;
- правила определяют, какие колонки показать в таблице.

Самые важные файлы:

- `src/domain/task-selectors.js` - выбирает задачи, домены, атрибуты и собирает строки таблицы;
- `src/domain/filters.js` - проверяет, подходит ли задача под выбранные фильтры;
- `src/domain/columns.js` - определяет, какие колонки показывать.

Если коротко:

- `src/data` - что есть;
- `src/domain` - как из этого выбрать нужное;
- интерфейс - как это показать пользователю.

## 8. Как задача превращается в строку таблицы

Внутри данные задачи хранятся не совсем так, как они выглядят в таблице.

Есть сама задача:

- номер;
- название;
- домен;
- статус;
- исполнитель;
- даты.

И отдельно есть значения ее атрибутов:

- тип документа;
- контрагент;
- сумма;
- SLA;
- объект;
- критичность;
- поставщик и так далее.

Перед показом таблицы система собирает это вместе.

Можно представить это как карточку и наклейки:

- карточка - это сама задача;
- наклейки - это дополнительные атрибуты;
- перед показом таблицы система приклеивает нужные наклейки к карточке;
- получившаяся карточка становится строкой таблицы.

Поэтому таблица получает уже готовые строки: номер, название, статус, домен и все нужные значения.

## 9. Как работают фильтры

Фильтры нужны, чтобы уменьшить список задач.

Например, пользователь может отфильтровать задачи по:

- номеру;
- названию;
- типу документа;
- контрагенту;
- сумме;
- бизнес-юниту;
- SLA;
- объекту;
- критичности;
- срокам.

Логика простая:

1. Система берет задачу.
2. Проверяет выбранные фильтры.
3. Если задача подходит под все активные фильтры, она остается в таблице.
4. Если не подходит хотя бы под один важный фильтр, она скрывается.

Например:

- выбран контрагент "ООО СтройПроект";
- задача имеет другого контрагента;
- значит эта задача не показывается.

Или:

- выбран диапазон суммы от 1 000 000 до 5 000 000;
- задача имеет сумму 700 000;
- значит она не попадает в таблицу.

## 10. Почему фильтры зависят от домена

У разных бизнес-доменов разные признаки.

Например:

- у строительных договоров есть "Тип документа", "Контрагент", "Сумма";
- у закупок ТМЦ есть "Бизнес-юнит", "SLA";
- у поручений СД есть "Объект", "Критичность".

Поэтому не каждый фильтр одинаково полезен во всех доменах.

Если пользователь смотрит "Закупки ТМЦ", фильтр по "Типу документа" может быть не главным. А если он смотрит "Строительные договоры", такой фильтр уже важен.

В режиме "Все задачи" система может использовать общий набор фильтров, потому что там смешаны задачи из разных доменов.

## 11. Как работает поиск

Поиск - это быстрый фильтр по тексту.

Он смотрит не только в одно поле, а по разным значениям строки задачи.

Например, если в строке есть:

- номер задачи;
- название;
- контрагент;
- объект;
- исполнитель;
- другой текстовый признак,

поиск может найти задачу по этим значениям.

Проще: поиск работает как быстрый способ найти задачу по тому, что пользователь помнит.

## 12. Как работают колонки

Колонки отвечают за то, какие данные пользователь видит в таблице.

Есть важное различие:

**данные задачи** и **видимые колонки** - это не одно и то же.

У задачи может быть больше данных, чем сейчас видно на экране.

Например, у задачи может быть:

- номер;
- название;
- контрагент;
- сумма;
- юридическое лицо;
- ставка НДС;
- ссылка на скан;
- условия оплаты.

Но в таблице прямо сейчас могут быть видны только:

- номер;
- название;
- контрагент;
- сумма.

Остальные данные не удалены. Они просто скрыты.

Настройки колонок позволяют выбрать, что показывать.

## 13. Почему колонки разные в разных доменах

У каждого бизнес-домена свой набор важных признаков.

Для строительных договоров важны одни колонки:

- тип документа;
- контрагент;
- сумма;
- дата создания;
- автор;
- проект.

Для закупок ТМЦ важны другие:

- бизнес-юнит;
- подразделение;
- SLA;
- срок;
- исполнитель.

Для поручений СД важны третьи:

- тип замечания;
- объект;
- критичность;
- этап;
- дата проверки.

Поэтому при переключении домена таблица меняет набор колонок.

Это нормально: таблица подстраивается под смысл выбранной рабочей области.

## 14. Что происходит в режиме "Все задачи"

Режим "Все задачи" особенный.

В нем смешиваются задачи из разных доменов, доступных текущей роли пользователя.

Проблема в том, что у разных доменов разные колонки. Например:

- у договоров есть "Сумма";
- у закупок есть "SLA";
- у поручений есть "Критичность".

Поэтому в режиме "Все задачи" система собирает общий набор колонок из доступных доменов.

Если у какой-то задачи нет значения для конкретной колонки, ячейка просто остается пустой.

Например:

- колонка "SLA" есть у закупок;
- у задачи по строительному договору SLA может не быть;
- в этой строке ячейка SLA будет пустой.

Это не ошибка. Это следствие того, что в общей таблице смешаны разные типы задач.

## 15. Что такое библиотека атрибутов

Библиотека атрибутов - это набор дополнительных полей, которые есть у домена, но не показаны в таблице сразу.

Можно представить это как ящик с запасными колонками.

Например, для строительных договоров в таблице по умолчанию могут быть видны:

- номер;
- название;
- тип документа;
- контрагент;
- сумма.

А в библиотеке могут лежать дополнительные атрибуты:

- юридическое лицо;
- ставка НДС;
- валюта договора;
- дата начала;
- дата окончания;
- ссылка на скан;
- условия оплаты.

Если пользователю нужно увидеть одно из этих полей, он добавляет его из библиотеки в таблицу.

## 16. Зачем нужна библиотека атрибутов

Если сразу показать все возможные колонки, таблица станет слишком большой и неудобной.

Библиотека атрибутов решает эту проблему:

- самые важные поля видны сразу;
- дополнительные поля не мешают;
- пользователь может добавить нужное поле, когда оно ему понадобится.

Это дает баланс:

- таблица не перегружена;
- данные не потеряны;
- пользователь может настроить вид под свою задачу.

Важно: добавление атрибута из библиотеки не создает новый атрибут в системе. Он уже существует. Пользователь просто делает его видимым в таблице.

## 17. Чем отличается атрибут от колонки

Это важный момент.

**Атрибут** - это признак задачи в системе.

**Колонка** - это отображение этого признака в таблице.

Например:

- "Контрагент" как атрибут означает, что у задачи есть такое значение;
- "Контрагент" как колонка означает, что это значение сейчас видно в таблице.

Атрибут может существовать, даже если колонка скрыта.

Поэтому скрытие колонки не удаляет данные. Оно только убирает их с экрана.

## 18. Как настройки колонок связаны с библиотекой атрибутов

Настройки колонок управляют тем, что пользователь видит.

В них есть:

- текущие видимые колонки;
- дополнительные атрибуты из библиотеки.

Когда пользователь добавляет атрибут из библиотеки:

1. система проверяет, что такой атрибут доступен для текущего домена;
2. добавляет его в список колонок;
3. таблица начинает показывать новое поле;
4. исходные данные задачи при этом не меняются.

Когда пользователь убирает колонку:

1. колонка исчезает из таблицы;
2. данные задачи остаются на месте;
3. атрибут может снова быть доступен в библиотеке.

## 19. Где находится внешний вид проекта

За внешний вид отвечает папка:

`src/styles`

Там описано, как выглядят:

- основная страница;
- левая панель;
- правая рабочая область;
- таблица;
- кнопки;
- поля поиска;
- фильтры;
- настройки колонок.

Если данные отвечают на вопрос "что показать", то стили отвечают на вопрос "как это должно выглядеть".

## 20. Где находится поведение интерфейса

За действия пользователя отвечает папка:

`src/features`

Там лежат части, которые управляют конкретными зонами интерфейса:

- фильтры;
- настройки колонок;
- боковая панель;
- таблица задач;
- группировка;
- выбор задач;
- пагинация.

Например:

- пользователь открыл фильтры - работает часть про фильтры;
- пользователь добавил колонку - работает часть про колонки;
- пользователь выбрал домен слева - работает часть про боковую панель;
- пользователь переключил очередь - обновляется список задач.

## 21. Как проект запускается

Главная страница проекта:

`index.html`

Она содержит основу интерфейса.

Главный файл запуска:

`src/main.js`

Он включает разные части интерфейса:

- навигацию;
- очереди;
- поиск;
- фильтры;
- таблицу;
- настройки колонок;
- дополнительные действия.

Можно представить `src/main.js` как человека, который утром включает свет во всех комнатах офиса: запускает нужные блоки, чтобы интерфейс начал работать.

## 22. Что такое `dist`

Папка `dist` - это готовая собранная версия проекта.

Если `src` - это кухня, где готовят интерфейс, то `dist` - это уже готовое блюдо, которое можно открыть в браузере.

Обычно правки делают не в `dist`, а в исходных файлах проекта. Потом проект собирается, и новая готовая версия попадает в `dist`.

## 23. Простая схема связей

Вот как все связано:

```text
Данные задач
  ↓
Правила выбора данных
  ↓
Фильтры, очереди, домены, колонки
  ↓
Готовые строки таблицы
  ↓
Интерфейс показывает результат пользователю
```

Или еще проще:

```text
Что есть в системе
  → что выбрал пользователь
  → что подходит под выбор
  → что показать в таблице
```

## 24. Самое важное, что нужно запомнить

1. Главный объект проекта - задача.

2. Задачи разделены по бизнес-доменам.

3. Статусы над таблицей показывают, в какой очереди находятся задачи.

4. Фильтры уменьшают список задач по выбранным признакам.

5. Колонки управляют тем, какие признаки видны в таблице.

6. Библиотека атрибутов хранит дополнительные поля, которые можно добавить в таблицу.

7. Данные лежат в `src/data`.

8. Правила работы с данными лежат в `src/domain`.

9. Поведение интерфейса лежит в `src/features`.

10. Внешний вид лежит в `src/styles`.

## 25. Главное отличие текущей структуры от хаотичного прототипа

Раньше проект мог быть похож на один большой лист, где все смешано: данные, таблица, фильтры, колонки и поведение кнопок.

Сейчас структура стала понятнее:

- данные лежат отдельно;
- правила работы с задачами лежат отдельно;
- интерфейсные действия лежат отдельно;
- внешний вид лежит отдельно.

Это важно для развития проекта.

Если в будущем нужно будет добавить создание задачи, редактирование, роли, новые домены или подключение к настоящим данным, проект уже имеет основу, куда это можно аккуратно встроить.

## Короткий итог

S-Tracker работает так:

пользователь выбирает рабочее направление и очередь, система берет подходящие задачи, применяет поиск и фильтры, собирает нужные колонки и показывает результат в таблице.

Бизнес-домены отвечают за рабочие области. Очереди отвечают за положение задач в рабочем маршруте. Статусы отвечают за состояние самой задачи и показываются цветными тегами. Фильтры отвечают за сужение списка. Колонки отвечают за видимость данных. Библиотека атрибутов отвечает за дополнительные поля, которые можно добавить в таблицу.

Главная ценность текущей структуры в том, что проект уже не просто показывает экран, а хранит понятную модель: какие задачи есть, какие признаки у них есть, как они связаны с доменами и как пользователь управляет их отображением.
