# Стандарт документации интерфейса

## 1. Назначение

Документация в `docs/` описывает текущий работающий прототип как спецификацию интерфейса.

Один и тот же документ должен быть понятен команде и пригоден как самостоятельный контекст для AI-агента, который не получает исходный проект.

## 2. Главный принцип

Документация должна быть чертежом текущего интерфейса.

Текст должен отвечать на вопрос: что именно агент должен построить, изменить или сохранить в интерфейсе.

Документ описывает только фактические элементы, состояния и поведение прототипа. Будущие идеи и желаемые улучшения фиксируются отдельно от текущего поведения.

## 3. Правило максимальной конкретики

Это главное правило приемки текста. Любое утверждение, которое не проходит это правило, считается непригодным для компонентной спецификации.

Каждое утверждение должно содержать четыре части:

| Часть требования | Что указывать |
|---|---|
| Элемент | Видимое название зоны, блока, кнопки, поля, параметра, строки, тега, состояния или соседней области. |
| Условие | Действие пользователя, выбранный режим, активный параметр, пустое состояние, ошибка или другое точное условие. |
| Изменение | Видимость, текст, число, значение, доступность, выбранность, активность, состав списка или результат в соседней области. |
| Итог | Конкретное значение или состояние: `видим`, `скрыт`, `активен`, `неактивен`, `выбран`, `не выбран`, пустое значение, число, текст или точный набор элементов. |

Рабочая формула предложения:

`При [условии] элемент [название] получает [конкретное состояние или значение].`

Слова риска допустимы только внутри полного требования:

| Слово риска | Как использовать корректно |
|---|---|
| `обновляется`, `пересчитывается`, `меняется` | Указать, какой элемент получает какое новое значение. |
| `отображается`, `появляется`, `скрывается` | Указать, где элемент находится, при каком условии и что остается видимым. |
| `обрабатывает` | Заменить на видимый результат или правило состояния. |
| `управляет`, `обеспечивает`, `позволяет`, `отвечает за` | Заменить на конкретное действие и итог интерфейса. |
| `служит для`, `используется для`, `дает возможность` | Заменить на условие, действие и результат. |

Примеры:

| Слабо | Конкретно |
|---|---|
| `Кнопка позволяет сбросить фильтры.` | `При клике по кнопке Сбросить всё все параметры фильтрации получают пустое значение; правая панель закрывается.` |
| `Поле используется для поиска.` | `После ввода текста в поле Поиск по задачам список задач показывает только задачи, где номер или наименование содержит введенный текст.` |
| `Счетчик обновляется.` | `После выбора первого значения в параметре Контрагент число в строке Выполнено настроек: N устанавливается в 1.` |
| `Доменные блоки скрываются.` | `При выборе бизнес-домена Закупки ТМЦ в панели фильтрации виден блок Закупки ТМЦ; блоки Строительные договоры и Поручения СД не видны.` |

Проверка приемки:

- можно назвать элемент, который меняется;
- можно назвать условие изменения;
- можно назвать итоговое значение или состояние;
- можно проверить результат по интерфейсу, состоянию компонента или правилу данных.

Если хотя бы один пункт не выполняется, текст нужно переписать.

### Механика устранения абстракций

Абстрактная формулировка заменяется проверяемым интерфейсным правилом. Проверяется не только наличие слов риска, а смысл утверждения.

| Тип утверждения | Что должно быть вместо абстракции |
|---|---|
| Расположение | Пространственный контракт: область, привязка, условие видимости, слой интерфейса, влияние на соседние области. |
| Действие пользователя | Действие, исходное состояние, точный результат в текущем компоненте и при необходимости результат в соседнем компоненте. |
| Состояние | Условие входа, видимое состояние, условие выхода или сброса. |
| Связь компонентов | Компонент-источник, действие или состояние, передаваемый результат, компонент-получатель, видимый итог. |
| Данные | Источник значения на уровне интерфейса, место использования, видимое значение или правило отбора. |
| Ограничение | Что компонент не создает, не открывает, не изменяет или не сбрасывает. |

Порядок переписывания абстрактного предложения:

1. Определить конкретный элемент или область интерфейса.
2. Определить тип утверждения из таблицы выше.
3. Заменить общую фразу на проверяемое правило соответствующего типа.
4. Удалить вводную или объяснительную часть, если она не добавляет условие, состояние или итог.
5. Проверить, что правило можно подтвердить в интерфейсе без чтения исходного кода.

Пример:

| Абстрактно | Инженерно |
|---|---|
| `Фильтры влияют на список задач.` | `После клика по кнопке Применить список задач показывает только задачи, которые проходят все активные параметры фильтрации.` |
| `Панель управляет массовыми действиями.` | `При выборе минимум одной задачи плавающая панель массовых действий видна поверх основного интерфейса; счетчик панели показывает количество выбранных задач.` |
| `Кнопка связана с панелью.` | `При клике по кнопке Фильтры в панели инструментов открывается правая панель фильтрации.` |

## 4. Иерархия правил

| Приоритет | Правило | Как применять |
|---|---|---|
| 1 | Максимальная конкретика | Сначала проверить каждое утверждение по правилу из раздела 3. |
| 2 | Соответствие текущему прототипу | Описывать фактические элементы, состояния и действия. |
| 3 | Переносимость | Писать спецификацию интерфейса без привязки к коду текущего проекта. |
| 4 | Самодостаточность в зоне ответственности | Каждый файл должен раскрывать свою часть компонента без поиска по проекту и без полного дублирования соседних файлов. |
| 5 | Компактность без потери правил | Убирать повторы и вводные фразы; сохранять условия, состояния и результаты действий. |
| 6 | Единая структура | Для крупных компонентов использовать `01-structure.md`, `02-behavior.md`, `03-data-states-rules.md`. |

Если правила конфликтуют, приоритет выше у более ранней строки таблицы.

## 5. Соответствие прототипу и переносимость

Код прототипа можно использовать для проверки фактического поведения. Технические детали кода в компонентную спецификацию не переносятся.

| В спецификации компонента требуется писать | Вместо |
|---|---|
| Видимые зоны, элементы и состояния | DOM, селекторы, классы, технические идентификаторы. |
| Действие пользователя и результат интерфейса | Названия функций, скриптов и модулей. |
| Роль данных и видимое правило | Пути файлов текущего проекта. |
| Условия, состояния и значения | Внутренние переменные и технические события. |

Карты связи документации с кодом размещаются только в служебных документах сопровождения, не в спецификациях компонентов.

### Инварианты и текущее наполнение прототипа

Спецификация должна отделять постоянное правило интерфейса от текущего наполнения прототипа.

| Тип информации | Как писать | Что не фиксировать как постоянное правило |
|---|---|---|
| Инвариант интерфейса | Описывать как обязательное правило поведения или структуры. | Не смешивать с текущими примерами данных. |
| Текущее наполнение прототипа | Указывать как `в текущем прототипе` или `текущий видимый набор`. | Не превращать текущий набор в полный список системы. |
| Роль-зависимый список | Указывать условие показа и правило изменения набора. | Не фиксировать постоянное количество, полный состав и постоянный порядок элементов. |

Пример: пункт `Все задачи` расположен перед списком бизнес-доменов. Это инвариант текущей структуры навигации.

Пример: `Строительный договор (ECM)`, `Закупки ТМЦ (Трекер)`, `Поручения СД` - текущий набор бизнес-доменов для одной роли. Для другой роли набор и порядок бизнес-доменов могут отличаться.

### Роль-зависимые списки

Если состав элементов зависит от роли, документация должна фиксировать тип элемента и правило показа, а не жесткий перечень позиций.

| Нужно указать | Требование |
|---|---|
| Тип элемента | Например: `Бизнес-домен`, `доступный атрибут`, `доменный блок фильтрации`. |
| Условие показа | Например: элемент виден, если доступен текущей роли или выбранной области задач. |
| Текущий пример | Перечислить значения текущего прототипа только как текущий видимый набор. |
| Правило порядка | Указать, кто задает порядок: роль, выбранная область, настройки пользователя или постоянное правило интерфейса. |

Не использовать формулировки `первый бизнес-домен`, `следующий домен`, `под предыдущим доменом`, если порядок не является постоянным правилом интерфейса.

### Динамически создаваемые блоки

Если блок появляется после действия пользователя, документация должна фиксировать источник действия, место появления результата и владельца видимого состояния.

| Нужно указать | Требование |
|---|---|
| Источник действия | Компонент и элемент, где пользователь запускает создание блока. |
| Место результата | Компонент и область, где появляется созданный блок. |
| Владелец состояния | Компонент, который показывает блок и дает дальнейшие действия с ним. |
| Дальнейшее влияние | Как выбор или изменение созданного блока влияет на соседние компоненты. |

Пример: действие группировки находится в заголовке колонки табличного списка задач; видимый блок группировки появляется в левой навигации; левая навигация владеет раскрытием, выбором значения, удалением значения и сбросом группировки.

## 6. Что фиксировать в компоненте

Компонентная спецификация должна содержать:

- каркас и состав компонента;
- внешние связи компонента;
- границу родительского компонента, если он объединяет дочерние зоны или компоненты;
- дочерние компоненты с ссылками на их собственные спецификации;
- контракты взаимодействия с соседними компонентами;
- инварианты компонента отдельно от текущего наполнения прототипа;
- термины компонента, если бытовое значение слова может привести к неверной реализации;
- базовое состояние компонента до пользовательских изменений;
- роль-зависимые наборы элементов и правила их показа;
- динамически создаваемые блоки, если они появляются в компоненте после внешнего действия;
- видимые области и элементы;
- точки входа;
- состояния элементов и условия их появления;
- действия пользователя и конкретную реакцию интерфейса;
- данные и правила, которые влияют на видимый результат;
- соседние области, которые задают условия или получают результат.

Спецификация описывает UI/UX: видимые состояния, правила интерфейса, пользовательские действия и результат в соседних областях.

Спецификация не описывает backend: базы данных, API, серверные методы, авторизацию, хранение на сервере и техническую схему обмена данными.

Стандартные стили корпоративной библиотеки не описываются: палитра, типографика, декоративные детали, базовые состояния кнопок, полей и селектов.

Размер, цвет, расположение или визуальная индикация фиксируются, если они являются частью поведения или различают состояние.

### Пространственный контракт расположения

Расположение элемента описывается без CSS-свойств. Спецификация должна фиксировать интерфейсное правило размещения, а не технические параметры верстки.

| Что фиксировать | Требование |
|---|---|
| Область размещения | В какой видимой области интерфейса находится элемент. |
| Привязка | Относительно какой области, строки, панели, заголовка, карточки или экрана расположен элемент. |
| Условие видимости | При каком режиме, состоянии или действии пользователя элемент виден или скрыт. |
| Слой интерфейса | Элемент находится внутри потока компонента, поверх интерфейса, в модальном слое, в drawer или в закрепленной области. |
| Влияние на соседние области | Какие области элемент не сдвигает, не перекрывает или перекрывает как часть своего сценария. |
| Точка запуска | Какое действие или состояние приводит к появлению элемента, если элемент не виден постоянно. |
| Владелец состояния | Какой компонент определяет видимость, выбранность, раскрытие или закрытие элемента. |

Не указывать `margin`, `padding`, `top`, `bottom`, `left`, `right`, `z-index`, размеры, сеточные значения и классы, если эти значения не являются продуктовым правилом интерфейса.

Пример:

| Слабо | Конкретно |
|---|---|
| `Панель находится внизу экрана.` | `При выборе минимум одной задачи плавающая панель массовых действий видна поверх основного интерфейса в нижней центральной области экрана. Панель не занимает место в списке задач и не сдвигает строки, карточки, пагинацию, панель инструментов и левую навигацию.` |

### Функциональные иконки и визуальные обозначения

Если визуальный знак запускает действие, он описывается как управляющий элемент, а не как отдельный декоративный символ.

| В интерфейсе | Как писать в спецификации |
|---|---|
| Кликабельный плюс | `кнопка действия с иконкой плюса` |
| Кликабельная стрелка назад | `кнопка действия с иконкой стрелки назад` |
| Кликабельный крестик закрытия | `кнопка закрытия с иконкой крестика` |
| Некликабельная иконка состояния | `индикатор с иконкой ...` или `иконка ... как признак состояния` |
| Просто визуальный символ без действия и состояния | Не описывать отдельно, если он не влияет на сценарий. |

Запрещено писать только название знака, если знак является действием: `нажимает плюс`, `клик по крестику`, `строка показывает стрелку`.

Нужно указывать тип сущности и результат: `нажимает кнопку действия с иконкой плюса`; `строка показывает кнопку действия с иконкой стрелки назад`; `бейдж показывает иконку задачи как часть счетчика`.

Модальное окно, drawer, overlay и другой вспомогательный слой описываются внутри компонента, который открывает этот слой и использует его в пользовательском пути.

Toast-уведомление, пустое состояние, disabled/active-индикация и ошибка описываются внутри компонента, где пользователь видит это состояние или запускает действие, которое приводит к этому состоянию.

Отдельная спецификация для вспомогательного слоя создается только при двух условиях:

- слой имеет самостоятельный пользовательский сценарий;
- файлы этого слоя будут передаваться AI-агенту как отдельный контекст конкретной итерации.

Общие правила паттерна не выносятся в отдельную компонентную спецификацию. Если правило нужно всем будущим документам, оно фиксируется в этом стандарте, а не в `docs/components`.

Отдельная спецификация обратной связи не создается. Если toast появляется после копирования email, он описывается в компоненте списка задач. Если пустое состояние появляется после фильтрации, оно описывается в фильтрации и списке задач. Если кнопка получает `disabled`, это состояние описывается в компоненте этой кнопки или панели.

Если в прототипе есть разметка или визуальный слой без подтвержденного пользовательского действия, этот слой не описывается как действующий компонент. Такой слой фиксируется как предмет финальной сверки: его нужно либо подключить к пользовательскому сценарию и описать в компоненте-владельце, либо удалить как legacy после проверки влияния на прототип.

### Контракты взаимодействия компонентов

Контракт взаимодействия нужен, если действие пользователя начинается в одном компоненте, а видимый результат появляется или используется в другом компоненте.

Контракт не описывает внутренний код связи. Контракт фиксирует только ответственность компонентов и видимый результат.

| Что фиксировать | Требование |
|---|---|
| Компонент-источник действия | Где пользователь нажимает, выбирает или вводит значение. |
| Компонент-получатель результата | Где появляется новый блок, состояние, список, счетчик или выбранное значение. |
| Владелец состояния | Какой компонент показывает состояние и дает пользователю дальнейшие действия с ним. |
| Условие запуска | При каком видимом состоянии действие доступно. |
| Видимый результат | Какой элемент получает какое состояние, текст, список или значение. |
| Ограничение ответственности | Что текущий компонент не создает, не открывает и не изменяет. |

Формат контракта:

| Связанный компонент | Роль связанного компонента | Условие | Результат в текущем компоненте | Результат в связанном компоненте |
|---|---|---|---|---|

Если компонент только показывает результат действия из другого компонента, в его спецификации нужно указать внешний источник действия и внутренний результат.

Если компонент запускает изменение в другом компоненте, в его спецификации нужно указать локальное действие пользователя и внешний видимый результат.

Локальный контракт компонента размещается в `03-data-states-rules.md`. В `02-behavior.md` допускается только короткая фиксация внешнего результата действия, если без нее пользовательский сценарий неполный.

`02-behavior.md` отвечает на вопрос `что сделал пользователь и что увидел`. `03-data-states-rules.md` отвечает на вопрос `кто источник действия, кто получатель результата, кто владелец состояния и какое правило действует дальше`.

Пример для группировки:

| Компонент | Роль |
|---|---|
| Табличный список задач | Содержит действие группировки в заголовке группируемой колонки. |
| Левая навигация | Показывает созданную вложенную группировку под `Все задачи` или бизнес-доменом. |
| Список задач | После выбора значения группировки показывает задачи выбранного значения. |

Текст `компоненты связаны`, `компонент влияет на другой компонент`, `синхронизируется с другим компонентом` не является контрактом. Нужно указывать источник действия, получатель результата, владельца состояния и конкретный видимый результат.

### Интеграционный контекст связанных компонентов

Если задача состоит не в создании компонента, а в связывании уже описанных компонентов, агент не должен получать полные комплекты `01/02/03` всех связанных компонентов по умолчанию.

Для интеграционной задачи основной контекст формируется из контрактных частей `03-data-states-rules.md` связанных компонентов.

| Ситуация | Какой контекст использовать |
|---|---|
| Нужно связать два компонента по состоянию или результату действия | `03-data-states-rules.md` компонента-источника и `03-data-states-rules.md` компонента-получателя. |
| Неясно, где находится элемент, который запускает связь | Добавить только нужный `01-structure.md` компонента-источника. |
| Неясно, где появляется результат связи | Добавить только нужный `01-structure.md` компонента-получателя. |
| Связь зависит от конкретного действия пользователя | Добавить соответствующий раздел `02-behavior.md` компонента-источника. |
| Связь проходит через три и более компонента | Передавать только `03` этих компонентов и минимальные фрагменты `01` или `02`, без полных комплектов всех компонентов. |

Формула: компонент строится по `01/02/03`; связь компонентов реализуется по контрактам из `03` связанных компонентов.

## 7. Состав комплекта

Крупный компонент описывается небольшим набором Markdown-файлов.

| Файл | Роль |
|---|---|
| `01-structure.md` | Видимая структура: раскрытие, каркас компонента, внешние связи, области, элементы, состояния отображения. |
| `02-behavior.md` | Действия пользователя, основной путь, реакции интерфейса. |
| `03-data-states-rules.md` | Интерфейсные данные, состояния и правила, которые действуют в нескольких сценариях и не дублируют `01` или `02`. |

Рекомендуемый размер комплекта: 3-5 файлов. Если компонент можно описать в трех файлах без потери смысла, использовать три файла.

Мелкие действия описываются внутри файла той области, к которой они относятся. Отдельный файл нужен только тогда, когда блок становится слишком большим для чтения агентом.

Модальные сценарии не создают отдельный компонент по умолчанию. Модальное окно сохранения пресета относится к компоненту пресетов; модальное окно перемещения в группу относится к компоненту массовых действий.

Состояния обратной связи не создают отдельный компонент по умолчанию. Toast, пустое состояние, disabled/active и ошибка относятся к компоненту, который создает это состояние.

## 8. Последовательность чтения

Нумерация файлов задает порядок передачи контекста агенту:

1. `01-structure.md`;
2. `02-behavior.md`;
3. `03-data-states-rules.md`;
4. дополнительные файлы, если компонент нельзя удержать в трех документах.

Каждый файл самодостаточен в своей зоне ответственности. Он не должен заставлять агента искать смысл в исходном проекте.

## 9. Файл `01-structure.md`

Роль: зафиксировать видимую структуру компонента.

Первый раздел `01-structure.md` выбирается по типу компонента.

| Тип компонента | Первый раздел | Когда использовать |
|---|---|---|
| Скрытый компонент | `Раскрытие компонента` | Компонент не виден по умолчанию и появляется после действия пользователя: правая панель, модальное окно, раскрываемая область. |
| Постоянно видимый компонент | `Расположение` | Компонент виден в рабочем интерфейсе без предварительного открытия: панель инструментов, левая навигация, список задач, пагинация. |
| Компонент с неочевидной зоной ответственности | `Граница компонента` | Есть риск включить в компонент соседние области, внешние условия, точки запуска или результат в другой области. |

Если компонент одновременно скрытый и имеет неочевидную зону ответственности, сначала описывать `Граница компонента`, затем `Раскрытие компонента`.

Если компонент постоянно видим и имеет неочевидную зону ответственности, сначала описывать `Граница компонента`, затем `Расположение`.

Должен содержать:

- точки входа;
- каркас и состав компонента;
- внешние связи компонента;
- области интерфейса;
- состав элементов;
- инварианты структуры и текущие примеры наполнения;
- роль-зависимые списки без ложной фиксации порядка;
- состояния отображения;
- условия видимости;
- соседние области, которые задают условия или получают результат.

Основной формат: таблицы.

| Таблица | Когда использовать |
|---|---|
| `Действие вне компонента / Раскрытие компонента` | Внешний вызов или открытие компонента. |
| `Часть компонента / Позиция / Состав` | Верхнеуровневый каркас компонента без повторения деталей из последующих разделов. |
| `Внешняя область / Связь с компонентом` | Области, которые не входят в компонент, но задают условия или получают результат. |
| `Элемент или зона / Где описано` | Граница родительского компонента, где нужно отделить внутренние зоны от дочерних и соседних компонентов. |
| `Зона внутри родителя / Где находится полная спецификация / Какое состояние родителя использует зона / Когда результат виден` | Родительский компонент с дочерними компонентами. |
| `Состояние / Видимые части / Недоступные части` | Закрытое, открытое, пустое и активное состояние. |
| `Точка входа / Условие показа / Целевая реакция интерфейса` | Входы в компонент. |
| `Зона / Состав / Условие показа` | Внутренние области компонента. |
| `Элемент / Условие показа / Видимое состояние` | Составные контролы. |
| `Параметр / Блок / Тип управления / Видимое значение до выбора` | Поля, параметры и настройки. |
| `Тип пункта / Условие показа / Видимый текст в текущем прототипе / Правило порядка / Результат выбора` | Роль-зависимые или динамические списки, где нельзя фиксировать постоянную последовательность элементов. |

`01-structure.md` не должен смешивать части компонента и внешние области в одной таблице через значения `Входит` и `Не входит`.

Если список элементов зависит от роли, выбранной области, настроек пользователя или данных, `01-structure.md` должен описывать типы элементов и правила показа. Текущие значения прототипа указываются только как пример видимого набора.

`01-structure.md` не должен использовать порядковые формулировки `первый`, `второй`, `следующий`, `под предыдущим`, если порядок не является постоянным правилом интерфейса.

### Граница компонента

Граница компонента описывает не связи вообще, а зону ответственности компонента.

Граница нужна, если соседний блок:

- запускает компонент, но не является его частью;
- задает условие для компонента, но не находится внутри него;
- получает результат действия компонента, но не управляется как внутренняя область компонента;
- влияет на состав данных или видимый результат, но имеет собственную спецификацию;
- визуально находится рядом с компонентом и может быть ошибочно включен в его состав.

Формат раздела:

| Объект | Статус относительно компонента | Правило |
|---|---|---|
| Внутренняя область компонента | Входит | Описывать состав и состояния внутри текущего файла. |
| Внешняя точка запуска | Не входит, открывает компонент | Описывать только факт запуска; состав и поведение точки запуска описывает ее собственный компонент. |
| Внешнее условие | Не входит, задает условие | Описывать только какое состояние или выбранное значение влияет на текущий компонент. |
| Соседняя область результата | Не входит, получает результат | Описывать только какой видимый результат появляется после действия текущего компонента. |

Пример для списка задач:

| Объект | Статус относительно списка задач | Правило |
|---|---|---|
| Табличный список и карточный список | Входят | Описываются как режимы отображения списка задач. |
| Поле `Поиск по задачам` | Не входит, задает условие | После ввода текста список задач показывает только подходящие задачи. Само поле описывается в панели инструментов. |
| Левая навигация бизнес-доменов | Не входит, задает условие | Выбранная область определяет набор задач, который попадает в список. Навигация описывается отдельно. |
| Панель фильтрации | Не входит, задает условие после `Применить` | Список задач показывает результат примененных параметров. Состав панели фильтрации описывается отдельно. |
| Настройки колонок | Не входят, задают результат после `Применить` | Табличный вид получает примененный набор динамических колонок. Сама панель настроек описывается отдельно. |

Не писать в границе общие фразы вроде `связано с компонентом`, `влияет на компонент`, `зависит от компонента`.
Всегда указывать конкретный тип отношения: `открывает`, `задает условие`, `получает результат`, `не участвует`.

### Родительские и дочерние компоненты

Родительский компонент нужен, если несколько зон используют общее состояние, общий футер действий, общее правило применения или общий пользовательский путь.

Дочерний компонент нужен, если зона имеет собственную структуру, поведение и правила состояния, которые можно передать отдельному агенту без потери смысла.

| Ситуация | Как документировать |
|---|---|
| Зона является частью родителя и не имеет самостоятельного сценария | Описывать в `01-structure.md` родителя. |
| Зона имеет собственную структуру, поведение и состояния | Создать дочернюю папку или дочерний комплект файлов и дать ссылку из родительского `01-structure.md`. |
| Зона использует состояние родителя | В родительском файле указать, какое состояние используется и когда результат становится видимым. |
| Зона только находится рядом визуально | Описывать как соседний компонент или внешнюю область, не как дочерний компонент. |

Родительский файл не должен подробно повторять дочерний компонент. Он фиксирует место дочерней зоны, связь с состоянием родителя и ссылку на полную спецификацию.

Если в таблице границы указан самостоятельный компонент, рядом должна быть ссылка на его спецификацию. Если объект не является UI-компонентом, это нужно указать явно: например, `Не отдельный UI-компонент; источник данных описан отдельно`.

### Функциональный блок вместо одного UI-компонента

Если папка в `docs/components/` описывает функциональность, которая собирает несколько UI-элементов и состояний, первый файл должен называть ее функциональным блоком, а не единым UI-компонентом.

| Что выделить | Что фиксировать |
|---|---|
| Главный UI блока | Видимый элемент, с которым пользователь выполняет основные действия. |
| Точки запуска | Элементы в соседних компонентах, после действия с которыми блок получает входное состояние. |
| Состояния соседних элементов | Какие состояния появляются вне главного UI блока и кто ими владеет. |
| Дочерний UI | Модальные окна, панели или дополнительные области, открываемые из главного UI. |
| Внешние получатели результата | Компоненты или области, где появляется результат действия блока. |

В `01-structure.md` функционального блока не писать, что все связанные элементы `входят в компонент`. Нужно указывать роль каждого объекта: `точка запуска`, `главный UI`, `элемент главного UI`, `дочерний UI`, `внешний получатель результата`, `внешнее условие`.

### Термины и базовые состояния

Если термин влияет на реализацию поведения, он должен иметь точное определение в компоненте или в `03-data-states-rules.md`.

Термин нужно определить, если:

- слово имеет бытовое значение, но в интерфейсе означает конкретное состояние;
- термин используется в нескольких файлах одного компонента;
- термин влияет на сохранение, сброс, применение, видимость или порядок элементов;
- неверная трактовка термина может изменить пользовательский сценарий.

Базовое состояние описывается отдельно от чернового, примененного и сохраненного состояния.

| Состояние | Что нужно указать |
|---|---|
| Базовое состояние | Какой минимальный или исходный набор видит пользователь до изменений. |
| Черновое состояние | Где временно хранятся изменения до применения. |
| Примененное состояние | Когда изменения становятся видимыми в основном интерфейсе. |
| Сохраненное состояние | Как пользовательский набор отличается от базового и чернового состояния. |

`01-structure.md` не пересказывает пользовательский сценарий и не описывает правила отбора, подсчета и применения.

## 10. Файл `02-behavior.md`

Роль: описать действия пользователя и реакцию интерфейса.

Должен содержать:

- один основной пользовательский путь;
- действия пользователя;
- условия доступности действий;
- конкретные результаты действий;
- вторичные механики в тексте или таблицах.

Основной пользовательский путь размещается в начале файла, показывает главный happy path и не включает редкие ветки, ошибки и исключения.

Если внутри компонента есть крупный самостоятельный блок поведения, его раздел должен отделять назначение, мини-путь и реакции интерфейса.

| Слой раздела | Что фиксировать | Формат |
|---|---|---|
| `Назначение` | Что пользователь получает от блока и какой результат появляется в интерфейсе. | Короткая таблица `Объект / Что получает пользователь`. |
| `Мини-путь пользователя` | Основную последовательность действий внутри этого блока без редких веток. | Нумерованный список на 3-7 шагов. |
| `Реакции интерфейса` | Что меняется после действия пользователя при конкретном исходном состоянии. | Таблица `Действие пользователя / Исходное состояние / Итог интерфейса`. |
| `Правила` | Постоянные правила блока, которые действуют вне одного клика. | Короткая таблица `Правило / Значение`. |

Мини-путь не заменяет основной пользовательский путь всего компонента. Мини-путь используется только тогда, когда один раздел внутри компонента имеет собственную понятную последовательность действий.

Назначение, мини-путь, реакции и правила не объединяются в одну таблицу. Таблица реакций не должна объяснять цель блока; она фиксирует только действие, исходное состояние и итог интерфейса.

Если действие пользователя создает результат в другом компоненте, `02-behavior.md` фиксирует этот результат коротко и без полной контрактной таблицы. Полный локальный контракт размещается в `03-data-states-rules.md`.

Пример: в поведении списка задач можно указать, что после клика по действию группировки в заголовке колонки в левой навигации появляется вложенная группировка. Владелец состояния и правила взаимодействия фиксируются в `03`.

## 11. Файл `03-data-states-rules.md`

Роль: описать интерфейсные данные, состояния и правила, которые дают видимый результат.

`03-data-states-rules.md` не является третьим пересказом компонента. Он фиксирует правила, которые нужны агенту для корректной UI/UX-логики после того, как структура уже описана в `01`, а действия пользователя уже описаны в `02`.

Должен содержать:

- интерфейсные состояния компонента;
- пустые, активные, выбранные, примененные и черновые значения;
- правила активности параметров и счетчиков;
- правила доступности, видимости и отбора, если они зависят от состояния;
- правила сохранения, сброса и применения состояния;
- контракты взаимодействия, если состояние компонента создается или используется соседним компонентом;
- входящие связи: какие состояния, значения или примененные результаты компонент принимает от соседних компонентов;
- исходящие связи: какие состояния, значения или видимые результаты компонент передает соседним компонентам;
- владельца состояния для динамически созданных блоков;
- правила для роль-зависимых наборов, если они влияют на видимый результат;
- результат для соседних областей, если результат возникает из правил состояния.

Не должен содержать:

- повтор состава зон, блоков и элементов из `01-structure.md`;
- повтор пользовательских сценариев, кликов и пошаговых действий из `02-behavior.md`;
- служебные разделы сопровождения документации вроде `Когда обновлять документ`;
- backend-описания: базы данных, API, серверное хранение, серверные статусы и техническую схему обмена;
- таблицы, где единственная цель - еще раз перечислить видимые элементы без нового правила состояния.

| Таблица | Когда использовать |
|---|---|
| `Состояние / Пустое значение / Активное значение / Видимый результат` | Интерфейсные состояния компонента. |
| `Состояние / Условие / Видимый результат` | Правила состояния. |
| `Параметр / Пустое значение / Активное значение / Участие в счетчике` | Правила активности параметров. |
| `Условие / Объект проходит / Объект исключается` | Правила отбора объектов. |
| `Область / Условие состояния / Видимый результат` | Влияние состояния компонента на соседнюю область. |
| `Тип связи / Источник / Передаваемый результат / Получатель / Видимый итог` | Входящие и исходящие связи компонента. |
| `Связанный компонент / Роль связанного компонента / Условие / Результат в текущем компоненте / Результат в связанном компоненте` | Контракты взаимодействия между компонентами. |

Если правило полностью описано в `02-behavior.md` как реакция на одно действие пользователя, в `03-data-states-rules.md` его повторять не нужно.

Если одно правило применяется в нескольких действиях, влияет на несколько областей или определяет состояние компонента вне конкретного клика, оно должно находиться в `03-data-states-rules.md`.

Если один компонент запускает действие, а другой компонент показывает созданный блок или состояние, `03-data-states-rules.md` должен явно указывать владельца состояния.

## 12. Таблицы и сценарии

Таблицы использовать для состава, условий, состояний, правил отбора и сравнения состояний.

Основной пользовательский путь описывать коротким нумерованным списком в начале `02-behavior.md`.

Таблицы остаются источником истины по поведению компонента.

Для ветвлений использовать таблицы:

| Формат таблицы | Когда использовать |
|---|---|
| `Действие пользователя / Условие / Итог интерфейса` | Реакции компонента на действия пользователя. |
| `Действие пользователя / Исходное состояние / Итог интерфейса` | Реакции компонента, где средняя колонка описывает состояние интерфейса перед действием пользователя. |
| `Ситуация / Условие / Видимый результат` | Нелинейные ветки, исключения и пустые состояния. |
| `Область / Условие / Результат` | Влияние компонента на соседние области. |

Диаграммы не использовать как источник правил компонентной спецификации.

## 13. Сквозные пользовательские пути `docs/user-flows/`

Роль: описать сценарий, который проходит через несколько компонентов и дает один пользовательский результат.

`docs/user-flows/` не заменяет компонентные спецификации. Flow фиксирует порядок передачи результата между компонентами. Структура, локальное поведение и правила состояния остаются в `docs/components/<component>/01-structure.md`, `02-behavior.md` и `03-data-states-rules.md`.

Flow нужен, если выполняется хотя бы одно условие:

| Условие | Что фиксирует flow |
|---|---|
| Пользователь начинает действие в одном компоненте, а итог видит в другом компоненте | Компонент-источник, передаваемый результат, компонент-получатель и видимый итог. |
| Сценарий проходит через две или больше самостоятельные области интерфейса | Последовательность перехода между областями без дублирования внутренней спецификации каждой области. |
| Нужно связать несколько уже описанных компонентов в один пользовательский путь | Общий порядок действий и межкомпонентные результаты. |
| Изменение поведения одного компонента меняет сквозной сценарий | Какой шаг сценария получает новое состояние или новый результат. |

Flow не нужен, если сценарий полностью описывается внутри одного компонента. Такой сценарий размещается в `02-behavior.md` этого компонента. Если меняется только правило состояния одного компонента, обновляется `03-data-states-rules.md`.

Каждый файл в `docs/user-flows/` должен содержать:

| Раздел | Что указать | Формат |
|---|---|---|
| `Зачем нужен сценарий` | Пользовательская цель и финальный результат сценария. | 1-2 предложения. |
| `Граница flow` | Начальное состояние, конечный результат, что не входит в flow. | Таблица `Граница / Значение`. |
| `Участники интерфейса` | Компоненты, участвующие в сценарии, и роль каждого компонента. | Таблица `Участник / Роль в сценарии / Источник деталей`. |
| `Основной сценарий` | Главный путь пользователя через компоненты. | Таблица `Шаг / Действие пользователя или событие / Компонент-источник / Передаваемый результат / Компонент-получатель / Видимый итог`. |
| `Изменения состояния` | Состояния, которые получают новое значение в ходе flow. | Таблица `Состояние / Владелец состояния / Когда меняется / Видимый результат`. |
| `Ветки` | Нелинейные варианты, если они меняют итог сценария. | Таблица `Ситуация / Условие / Видимый результат`. |
| `Границы детализации` | Что этот flow не описывает и где находится источник истины. | Таблица `Что не описывает flow / Где находится источник истины`. |
| `Когда обновлять документ` | Конкретные изменения, после которых flow должен быть проверен. | Короткий список. |

Правила написания flow:

- не описывать внутреннюю структуру компонента, если она уже находится в `01-structure.md`;
- не повторять полный локальный сценарий компонента из `02-behavior.md`;
- не повторять контракты состояния из `03-data-states-rules.md`;
- указывать только те соседние компоненты, которые участвуют в текущем сценарии;
- описывать результат как видимое изменение интерфейса, а не как техническую передачу данных;
- не использовать пути к коду, DOM, селекторы, функции и скрипты;
- не использовать диаграмму как источник правил; таблицы являются источником истины.

Если сценарий перенесен в `docs/components/<component>/02-behavior.md` и больше не проходит через несколько компонентов, файл в `docs/user-flows/` удаляется из действующего списка. Не создавать навигационные заглушки, которые только перенаправляют к компонентной спецификации.

`docs/user-flows/README.md` является только локальным индексом flow-файлов. Формат и правила написания flow задает этот раздел стандарта.

## 14. Связи между файлами

Файлы одного компонента должны описывать один и тот же интерфейс без противоречий.

Связи использовать точечно:

- из `01-structure.md` родительского компонента нужно ссылаться на дочерние компоненты и самостоятельные соседние компоненты, если они упомянуты в границе или составе;
- из `02-behavior.md` можно ссылаться на структуру, если действие зависит от конкретной области интерфейса;
- из `03-data-states-rules.md` можно ссылаться на поведение, если правило применяется после конкретного действия;
- в `README.md` компонента можно дать карту комплекта файлов.

Полное описание одной логики не дублируется в нескольких файлах. Во втором файле достаточно короткой привязки.

`03-data-states-rules.md` может ссылаться на действие из `02`, но должен фиксировать правило состояния, а не повторять последовательность кликов.

## 15. Проверка перед публикацией

| Проверка | Критерий |
|---|---|
| Максимальная конкретика | Каждое утверждение содержит элемент, условие, изменение и итог. |
| Абстракции устранены | Общие формулировки заменены проверяемыми правилами: расположение, действие, состояние, связь, данные или ограничение. |
| Соответствие прототипу | Описаны фактические элементы, состояния и действия. |
| Переносимость | Нет путей проекта, DOM, селекторов, функций, скриптов и технических идентификаторов. |
| Самодостаточность | Файл понятен без чтения исходного проекта и без полного дублирования соседних файлов. |
| Нет дублей | Одинаковая логика не пересказывается в нескольких местах. |
| Роль `03` | `03-data-states-rules.md` содержит правила состояния и не повторяет структуру из `01` или пользовательские пути из `02`. |
| Расположение описано контрактно | Для значимых областей указаны область размещения, привязка, условие видимости, слой интерфейса и влияние на соседние области без CSS-параметров. |
| Нет лишней стилистики | Стандартные стили корпоративной библиотеки не описаны, если они не влияют на поведение. |
| Инварианты отделены от примеров | Постоянное правило интерфейса не смешано с текущим набором данных прототипа. |
| Роль-зависимые списки описаны корректно | Не зафиксированы постоянное количество, полный состав и порядок элементов, если они зависят от роли. |
| Динамические блоки имеют владельца | Указано, где пользователь запускает действие, где появляется результат и какой компонент владеет состоянием. |
| Локальный контракт не продублирован | В `02` есть только внешний результат действия; полная контрактная таблица находится в `03`. |
| Интеграционный контекст не перегружен | Для связи компонентов используются контрактные части `03` связанных компонентов; полные комплекты `01/02/03` добавляются только при явной необходимости. |
| Граница компонента разобрана | Внутренние зоны, дочерние компоненты, соседние компоненты, внешние условия и данные не смешаны в одном списке без типа связи. |
| Дочерние компоненты имеют ссылки | Если самостоятельный компонент упомянут в родительской спецификации, рядом есть ссылка на его файл или комплект файлов. |
| Термины определены | Термины, влияющие на поведение, состояние, сохранение, сброс, применение, видимость или порядок, имеют точное определение. |
| Базовое состояние отделено | Исходное состояние до пользовательских изменений не смешано с черновым, примененным или сохраненным состоянием. |
| Функциональные иконки описаны как контролы | Кликабельный визуальный знак описан как кнопка или управляющий элемент с иконкой, а не как самостоятельный символ. |

### Связанное обновление документации

Если меняется код, спецификация или структура проекта, агент должен проверить логически связанные документы. Обновлять нужно только те области документов, на которые реально влияет изменение.

| Что изменилось | Какие документы проверить и при необходимости обновить |
|---|---|
| Видимая структура компонента, состав зон, порядок элементов, точки входа, границы | `docs/components/<component>/01-structure.md` |
| Действия пользователя, реакции интерфейса, открытие или закрытие областей, применение, сброс, удаление | `docs/components/<component>/02-behavior.md` |
| Состояния, счетчики, правила видимости, владельцы состояния, контракты между компонентами | `docs/components/<component>/03-data-states-rules.md` |
| Сквозной пользовательский путь через несколько компонентов | Соответствующий файл в `docs/user-flows/` |
| Модель задач, атрибутов, доменов, очередей, справочников или значений | `docs/components/<component>/03-data-states-rules.md`, `docs/entities/` при необходимости |
| Расположение логики, перенос файла, новый файл реализации или изменение связи компонента с кодом | `docs/component-code-map.md` |
| Физическая структура проекта, новая папка, удаленная папка, изменение роли папки | `docs/project-structure.md`, при необходимости `docs/README.md` |
| Правила подготовки документации или маршрут контекста для агентов | `docs/documentation-standards.md`, `docs/agent-context-routing.md`, при необходимости `docs/documentation-standards-audit.md` |
| Проведена новая проверка соответствия документации стандарту | `docs/documentation-standards-audit.md` |

`docs/documentation-standards-audit.md` не является журналом всех изменений кода. Он обновляется, когда меняется статус проверки документации: зафиксировано новое несоответствие, закрыто несоответствие, проведен новый аудит или изменены правила, по которым выполняется аудит.

Если меняется документация как спецификация будущего изменения, явно отделить это от текущего поведения.
