# Стандарт документации интерфейса

Этот стандарт описывает, как писать документацию в `docs/`, чтобы ее можно было использовать как спецификацию для команды и как контекст для AI-агента.

## Главный принцип

Документация должна быть чертежом текущего работающего прототипа.

Писать нужно только то, что реально есть в коде и интерфейсе. Не добавлять будущие идеи, желаемые улучшения и нереализованные состояния.

## Что описывать

В каждом компоненте фиксировать:

- где компонент находится;
- из каких областей и элементов состоит;
- какие элементы видит пользователь;
- что происходит при каждом действии пользователя;
- какие состояния есть у компонента;
- какие данные участвуют в работе;
- как компонент влияет на соседние области интерфейса.

## Что не описывать

Не описывать стили корпоративной библиотеки, если они приходят из Storybook или отдельного UI skill:

- палитры;
- типографику;
- декоративные детали;
- стандартные состояния базовых кнопок, полей и селектов.

Исключение: если размер, расположение или визуальная индикация являются частью поведения, их нужно описать как функциональное требование.

## Размер комплекта

Один компонент описывается небольшим количеством Markdown-файлов.

Рекомендуемый размер: 3-5 файлов на компонент.

Если компонент можно описать в 3 файлах без потери смысла, использовать 3 файла.

Один файл должен быть небольшим и самодостаточным, но детализация важнее искусственного сокращения.

Сокращать можно повторы, вводные фразы и общие рассуждения. Нельзя сокращать правила, условия, состояния и детали поведения, если без них агент может неверно реализовать компонент.

Не дробить документацию на микромеханики. Мелкие действия описывать внутри файла той области, к которой они относятся.

## Последовательность файлов

Нумерация показывает порядок чтения и порядок передачи контекста агенту.

Для сложного компонента предпочтительная логика такая:

1. структура интерфейса;
2. поведение пользователя и реакция интерфейса;
3. данные, состояния и правила.

Пример для фильтрации:

- `01-structure.md`;
- `02-behavior.md`;
- `03-data-states-rules.md`.

## Основной пользовательский путь

У каждого интерактивного компонента должен быть один основной пользовательский путь.

Основной путь описывает главную цель компонента для пользователя: от входа в компонент до результата, ради которого компонент существует.

Правила:

- основной путь размещается в `02-behavior.md` сразу после вводного описания;
- основной путь должен быть один;
- путь описывается короткой последовательностью шагов без редких веток и исключений;
- путь не заменяет таблицы поведения, а задает общую рамку сценария;
- малые сценарии не оформляются как отдельные пользовательские пути;
- частные действия, состояния и исключения описываются таблицами в соответствующих разделах `02-behavior.md` и правилами в `03-data-states-rules.md`.

Если компонент не является интерактивным, основной пользовательский путь можно не добавлять, но это должно быть очевидно из назначения компонента.

## Связи между файлами

Файлы не должны быть изолированными.

Если один файл зависит от другого, нужно прямо указать ссылку:

- из файла поведения можно ссылаться на структуру;
- из файла структуры можно ссылаться на правила состояний;
- из файла правил можно ссылаться на поведение применения и сброса.

Ссылка нужна только там, где она помогает понять, куда смотреть за соседней частью описания.

## Уровень конкретики

Писать инженерно и точно.

Плохо:

- `Фильтры позволяют управлять списком задач`.
- `Пользователь может настроить параметры`.
- `Компонент влияет на отображение`.

Хорошо:

- `Кнопка Фильтры находится в верхней панели правой рабочей области`.
- `При активных фильтрах иконка фильтра скрывается, а на ее месте показывается круглый индикатор со счетчиком`.
- `После Применить панель закрывается, список задач перерисовывается, счетчики левой панели пересчитываются`.

## Таблицы

Активно использовать таблицы, если они делают описание точнее и короче.

Таблицы обязательны или предпочтительны для:

- состава блоков и элементов;
- условий отображения;
- правил активного и пустого состояния;
- правил прохождения объектов через фильтры;
- сравнения состояний;
- связи интерфейсного элемента с ключом данных.

Если таблица начинает скрывать порядок действий пользователя, использовать текстовый сценарий.

## Повторы между файлами

Не дублировать полное описание одной и той же логики в нескольких файлах.

Если логика уже описана в одном файле, в соседнем файле оставить короткую ссылку на него.

Пример:

- структура элемента описана в `01-structure.md`;
- действие пользователя описано в `02-behavior.md`;
- правило подсчета, фильтрации или состояния описано в `03-data-states-rules.md`.

## Конкретные формулировки поведения

Не писать обобщенно: `обновляется`, `пересчитывается`, `меняется состояние`, если можно указать точный результат.

Плохо:

- `обновляется счетчик`;
- `пересчитывается видимость`;
- `обновляется состояние фильтра`.

Хорошо:

- `число в строке Выполнено настроек меняется на количество активных параметров`;
- `доменные блоки других бизнес-доменов получают display: none`;
- `массив выбранных значений параметра получает новое значение, а поле показывает тег выбранного значения`.

## Mermaid

Mermaid использовать только там, где диаграмма реально помогает понять последовательность.

Диаграмма должна показывать главный путь, а не все редкие ветки.

Если текстовое описание точнее и короче диаграммы, использовать текст.

## Документация и код

Документация должна соответствовать текущему прототипу.

В компонентных спецификациях для AI-агента основной раздел про данные и логику должен описывать не пути файлов текущего проекта, а зоны реализации:

- какие данные должны существовать;
- какое состояние нужно хранить;
- какие правила отбора, сборки или отображения должны быть реализованы;
- как компонент использует эти данные и правила.

Пути к файлам текущего S-Tracker можно указывать только вторичным справочным блоком, например `Текущая реализация в S-Tracker`.

Не делать колонку `Где находится` главным источником смысла, если файл предполагается передавать AI-агенту как самостоятельный контекст.

Перед изменением документации проверить:

- `index.html` - наличие элементов интерфейса;
- `src/features` и `src/ui` - поведение;
- `src/domain` и `src/data` - данные и правила;
- `src/styles` - визуальные состояния, если они являются частью поведения.

Если меняется код компонента, обновить соответствующий файл документации.

Если меняется документация как спецификация будущего изменения, явно отделить это от описания текущего поведения.
