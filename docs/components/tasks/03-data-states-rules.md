# Список задач: данные, состояния и правила

Файл описывает, из каких данных собирается список задач, какие состояния используются и по каким правилам задачи попадают в таблицу или карточки. Структура описана в `01-structure.md`, действия пользователя - в `02-behavior.md`.

## 1. Источники данных и зоны реализации

Этот раздел описывает не пути текущего проекта, а то, какие зоны кода должны быть реализованы или подключены, чтобы список задач работал.

| Что нужно списку | Что должно быть реализовано в коде | Как список это использует |
|---|---|---|
| Данные задач | Коллекция объектов `Task` с номером, названием, бизнес-доменом, очередью, статусом, автором, исполнителем, датами и флагами индикации | Из этих объектов собираются карточки и строки таблицы |
| Значения атрибутов | Механизм получения значения по паре `taskId` + `attributeKey` или готовое поле в плоской строке задачи | Заполняет доменные колонки, дополнительные колонки и скрытые атрибуты после добавления из библиотеки |
| Бизнес-домены и доступ роли | Список бизнес-доменов, доступных текущей роли пользователя | Определяет, какие задачи попадают в режим `Все задачи` и какие домены участвуют в объединенной таблице |
| Очереди | Состояние активной очереди и правило отбора задач по очереди | Ограничивает список задач выбранным сегментом над таблицей |
| Статусы задачи | Справочник статусов с системным значением, текстом и визуальным классом или цветом тега | Позволяет показать в карточке и таблице правильный текст и цветной тег статуса |
| Авторы и пользователи | Справочник пользователей с именем, email и аватаром | Позволяет показать автора задачи и скопировать email по клику |
| Колонки таблицы | Правило получения активных колонок для текущего контекста и пользовательских настроек | Определяет заголовки таблицы, порядок ячеек и закрепленные колонки |
| Фильтры и поиск | Функция проверки задачи на соответствие активным параметрам и поисковой строке | Исключает задачи, которые не должны отображаться в текущем списке |
| Навигационный контекст | Состояние выбранного пункта слева: все задачи, домен, группа или сгруппированное значение | Дополнительно ограничивает набор задач до нужной области |
| Пагинация | Состояние текущей страницы, размер страницы и правило нарезки массива задач | В интерфейс попадают только задачи текущей страницы |
| Выбор задач | Набор выбранных `taskId` и функции добавления, удаления и очистки выбора | Управляет чекбоксами, selected-состоянием и панелью массовых действий |
| Рендер списка | Функция, которая получает итоговый набор задач и выводит его в карточный и табличный контейнеры | Обновляет DOM, пустое состояние, счетчики, пагинацию и события после рендера |

## 2. Состояния, влияющие на список

| Состояние | Где хранится | Как влияет |
|---|---|---|
| `window.navigationContext` | Глобальное состояние | Определяет, какие домены, группы или сгруппированные значения показывать |
| Активная очередь | DOM: `.js-status-tabs .is-active` | Определяет очередь задач для списка |
| `window.activeFilters` | Глобальное состояние | Отсекает задачи по поиску и параметрам фильтров |
| `window.customColumnState` | Глобальное состояние | Меняет состав и порядок колонок таблицы |
| `window.paginationState` | Глобальное состояние | Определяет текущую страницу и размер страницы |
| `window.selectedTaskIds` | Глобальное состояние | Хранит выбранные задачи |
| `window.__taskDataMap` | Глобальный служебный кеш | Нужен для массовых действий по выбранным задачам |

## 3. Правило сборки списка

При каждом рендере список проходит одну и ту же цепочку.

| Порядок | Правило | Результат |
|---|---|---|
| 1 | Берется layout активной очереди | Система понимает, сколько задач брать по каждому бизнес-домену |
| 2 | Для каждой позиции создается плоская строка задачи | Задача получает значения атрибутов, статус, автора, флаги |
| 3 | Применяется левая навигация | Убираются задачи вне выбранного домена, группы или значения |
| 4 | Применяются активные фильтры и поиск | Убираются задачи, не прошедшие параметры |
| 5 | Задачи сортируются | Новые задачи идут выше обычных; просроченные идут выше обычных после новых |
| 6 | Применяется пагинация | В DOM попадают только задачи текущей страницы |
| 7 | Одни и те же задачи рендерятся в карточки и в строки таблицы | Пользователь видит активный режим отображения |

## 4. Правила навигационного контекста

| Тип контекста | Условие прохождения задачи |
|---|---|
| `all` | Задача проходит, если ее бизнес-домен доступен текущей роли |
| `system` с `id = all` | То же правило, что `all` |
| `system` с `id` бизнес-домена | `task.businessDomainId` должен совпадать с `id` |
| `grouped-value` | Задача должна принадлежать домену `id` и иметь значение `val` в атрибуте `attr` |
| `custom-group` | `task.id` должен входить в `taskIds` пользовательской группы |

Если `grouped-value.val` равен `Без значения`, проходят задачи, у которых соответствующий атрибут пустой, `null` или `undefined`.

## 5. Колонки таблицы

| Контекст | Какие колонки берутся |
|---|---|
| `Все задачи` | Объединенный набор дефолтных колонок всех доступных доменов |
| Конкретный бизнес-домен | Дефолтные колонки этого домена |
| Пользовательская группа | Объединенный набор колонок, если `id` группы не совпадает с бизнес-доменом |
| После применения настроек колонок | Фиксированные колонки остаются первыми, остальные берутся из `window.customColumnState` |

Фиксированные колонки:

| Ключ | Название | Правило |
|---|---|---|
| `cb` | Пустой заголовок | Всегда первая, не удаляется настройками |
| `id` | `№` | Всегда после чекбокса, не удаляется настройками |
| `title` | `Наименование` | Всегда после номера, не удаляется настройками |

## 6. Значения задачи в списке

| Ключ | Откуда берется значение | Как показывается |
|---|---|---|
| `id` | `Task.id` | Номер задачи |
| `title` | `Task.title` | Наименование с иконкой задачи |
| `sys` | `Task.businessDomainId` | Код домена в верхнем регистре |
| `status` | `Task.taskStatusId` через `getTaskStatus` | Цветной тег статуса |
| `author` | `Task.createdBy` через справочник пользователей | Автор с аватаром и email |
| `assignee` | `Task.assigneeId` или значение атрибута | Исполнитель |
| Даты | Значения атрибутов задачи | Дата с иконкой календаря |
| Остальные атрибуты | `TASK_ATTRIBUTE_VALUES` | Текстовое значение или `–`, если значения нет |

## 7. Соответствие данных карточке и строке таблицы

| Данные задачи | Карточный вид | Табличный вид |
|---|---|---|
| `id` | Показывается в основной зоне карточки | Показывается в колонке `№` |
| `title` | Показывается после иконки задачи, обрезается многоточием | Показывается в колонке `Наименование` после иконки задачи, обрезается многоточием |
| `taskStatusId` | Превращается в тег статуса в правой зоне карточки | Превращается в тег статуса в колонке `Статус`, если колонка активна |
| `createdBy` | Превращается в автора: аватар, имя, email | Превращается в автора в колонке `Автор`, если колонка активна |
| Date-атрибуты | Показывается одна дата, выбранная по правилам карточки | Показываются все активные date-колонки |
| `businessDomainId` | Напрямую не показывается в карточке | Может показываться как `sys` в объединенной таблице |
| Доменные атрибуты | Не показываются отдельно | Показываются в активных колонках таблицы |
| `flags.isNew` | Дает зеленую индикацию карточки | Дает зеленую индикацию строки |
| `flags.isOverdue` | Дает красную индикацию карточки и красную дату | Дает красную индикацию строки и красную дату |
| Наличие `task.id` в выбранных задачах | Делает чекбокс карточки активным | Делает чекбокс строки активным |

## 8. Статусы задачи

Статус задачи - это отдельный атрибут задачи, не очередь. Очереди описаны в `../queues/`.

| Статус | Системное значение | Класс тега |
|---|---|---|
| Новая | `new` | `ds-tag--task-status-new` |
| Назначена | `assigned` | `ds-tag--task-status-assigned` |
| В работе | `in_progress` | `ds-tag--task-status-in-progress` |
| Назначение снято | `assignment_removed` | `ds-tag--task-status-assignment-removed` |
| Завершена | `completed` | `ds-tag--task-status-completed` |
| Отменена | `cancelled` | `ds-tag--task-status-cancelled` |
| Отклонена | `rejected` | `ds-tag--task-status-rejected` |

Каждый статус отображается в виде контрастного цветного тега с белым текстом.

## 9. Визуальные состояния задачи

| Состояние | Откуда берется | Карточный вид | Табличный вид |
|---|---|---|---|
| Обычная задача | Нет флагов `isNew` и `isOverdue` | Белая карточка; при hover светло-серый фон | Белые ячейки; при hover светло-серый фон всех ячеек |
| Новая задача | `task.flags.isNew` | Зеленоватая строка и зеленая левая индикация | Зеленоватые ячейки и зеленая левая индикация в первой ячейке |
| Просроченная задача | `task.flags.isOverdue` | Розоватая строка и красная левая индикация | Розоватые ячейки и красная левая индикация в первой ячейке |
| Выбранная задача | `window.selectedTaskIds` содержит `task.id` | Чекбокс активен, карточка получает выбранное состояние | Чекбокс активен, строка получает выбранное состояние |

У просроченной задачи красным выделяется значение даты, которая считается сроком или контрольной датой. Красная обводка у тега статуса не используется.

## 10. Правило даты просрочки

| Бизнес-домен | Какая дата получает alert-состояние в таблице |
|---|---|
| `ecm` | `regDate` |
| `tracker` | `dueDate` |
| `control` | `checkDate` |
| `control` в объединенном режиме с колонкой `sys` | Может подсвечиваться `regDate` |

В карточном виде для просроченной задачи выбирается одна дата: `dueDate` для `tracker`, `checkDate` для `control`, иначе `regDate`.

## 11. Выбор задач

| Действие | Что меняется в данных |
|---|---|
| Выбор отдельной задачи | `task.id` добавляется в `window.selectedTaskIds` |
| Снятие выбора отдельной задачи | `task.id` удаляется из `window.selectedTaskIds` |
| Выбор всех в таблице | В `window.selectedTaskIds` добавляются все `id` видимых строк текущей страницы |
| Снятие всех видимых строк таблицы | Из `window.selectedTaskIds` удаляются все `id` видимых строк текущей страницы |
| Снятие выделения в плавающей панели | `window.selectedTaskIds` очищается полностью |

Состояние чекбокса в заголовке таблицы рассчитывается только по видимым строкам текущей страницы.

## 12. Пагинация

| Правило | Значение |
|---|---|
| Размер страницы | 20 задач |
| Если задач 0 | Текущая страница становится 1 |
| Если текущая страница больше количества страниц | Текущая страница уменьшается до последней доступной |
| Если задач не больше 20 | Пагинация скрывается |
| Если задач больше 20 | Пагинация показывается |

При переходе по страницам список перерисовывается с сохранением выбранной страницы. При смене очереди, домена или фильтров список обычно возвращается к первой странице.

## 13. События после рендера

После рендера списка вызывается событие `table-rendered`.

| Кто использует событие | Для чего |
|---|---|
| Выбор задач | Синхронизирует чекбокс выбора всех в заголовке таблицы |
| Табличный скролл | Может синхронизировать служебные элементы таблицы |

Также после рендера обновляются:

- количество задач в детальном счетчике, если он применим;
- пагинация;
- бейджи очередей;
- кеш `window.__taskDataMap` для выбранных задач.

## 14. Пустые значения

Если у задачи нет значения для активной колонки, в ячейке показывается `–`.

Это особенно важно для режима `Все задачи`: объединенная таблица может содержать колонки, которые есть не во всех бизнес-доменах.
