# Маршрут контекста для AI-агента и субагентов

## 1. Назначение

Этот документ задает правило выбора ограниченного пакета контекста для работы с документацией, интерфейсом и функциональностью S-Tracker.

Цель маршрута: агент должен читать только те документы, которые нужны для конкретной задачи, не смешивать границы компонентов и не угадывать поведение по соседним файлам.

Файл, папка или раздел проекта не считается рабочим контекстом только потому, что он существует в структуре проекта. Рабочим контекстом становится только тот документ, который явно выбран главным агентом для текущей задачи или добавлен после проверки причины расширения контекста.

## 2. Правило минимального рабочего контекста

Перед чтением документов главный агент определяет границу задачи и фиксирует рабочий пакет.

| Шаг | Что определить | Результат |
|---|---|---|
| 1 | Тип задачи | Документация, реализация, проверка качества, связь компонентов, изменение стандарта. |
| 2 | Целевая область | Один компонент, дочерняя зона, несколько связанных компонентов, сквозной пользовательский путь. |
| 3 | Минимальный пакет | Конкретные файлы, которые нужны для решения задачи. |
| 4 | Запрещенная область | Папки и документы, которые не читаются без отдельной причины. |
| 5 | Условие расширения | Какой конкретный факт должен быть непонятен, чтобы добавить новый файл. |

Правило работы:

- агент не читает всю папку `docs/`, `docs/components/` или `src/` для локальной задачи;
- агент не использует соседние компоненты как фоновые знания, если они не входят в выбранный пакет;
- агент не читает `PROJECT-OVERVIEW.md` для обычной компонентной задачи;
- агент не читает весь `docs/documentation-standards.md`, если достаточно применимых разделов;
- агент расширяет пакет только после определения конкретного вопроса, на который не отвечает текущий пакет;
- каждый добавленный документ должен иметь причину добавления и ожидаемый ответ.

Формула выбора контекста:

```text
минимальный пакет текущей задачи
+ только явно необходимые связанные документы
- все документы без прямой связи с задачей
```

Если задача сформулирована широко, главный агент сначала сужает ее до конкретной области работы. Если сузить область невозможно без решения пользователя, главный агент предлагает варианты границы задачи.

## 3. Базовый контекст главного агента

Главный агент перед планированием или реализацией выбирает минимальный базовый контекст. Не каждый документ из таблицы читается всегда.

| Документ | Когда читать | Зачем нужен |
|---|---|---|
| `AGENTS.md` | Всегда перед изменениями в проекте. | Постоянные правила проекта и ограничения на изменения. |
| `docs/README.md` | При работе с документацией, компонентами или маршрутом контекста. | Верхняя карта документации. |
| `docs/components/README.md` | При выборе компонентной спецификации или проверке состава компонентов. | Карта компонентных спецификаций. |
| `docs/documentation-standards.md` | При создании, проверке или изменении документации; читать только применимые разделы по разделу 4 этого маршрута. | Обязательные правила оформления и проверки документации. |
| `PROJECT-OVERVIEW.md` | Только при первичном входе в проект, продуктовых вопросах, неясности терминов или недостатке общего понимания S-Tracker. | Общее понимание продукта и назначения прототипа. |

`PROJECT-OVERVIEW.md` не входит в обычный пакет компонентной задачи и не передается субагенту по умолчанию.

Главный агент не передает субагенту всю папку `docs/` или всю папку `docs/components/`.

## 4. Маршрутизация разделов `Documentation Standards`

`docs/documentation-standards.md` является основным стандартом, но для обычной компонентной задачи агент не должен читать его целиком. Главный агент сначала определяет тип задачи, затем выбирает только нужные разделы стандарта.

| Тип задачи | Какие разделы `docs/documentation-standards.md` читать | Зачем нужны |
|---|---|---|
| Любая новая или изменяемая компонентная спецификация | 1 `Назначение`; 2 `Главный принцип`; 3 `Правило максимальной конкретики`; 5 `Соответствие прототипу и переносимость`; 7 `Состав комплекта`; 8 `Последовательность чтения`; 15 `Проверка перед публикацией` | Задает общий уровень конкретики, состав комплекта и финальную проверку. |
| Структура, состав зон, расположение, границы компонента | 3 `Правило максимальной конкретики`; 6 `Что фиксировать в компоненте`; 9 `Файл 01-structure.md`; 14 `Связи между файлами`; 15 `Проверка перед публикацией` | Помогает описать видимую структуру, не смешивать компонент с контекстом и проставить ссылки. |
| Граница компонента, соседние области, родительские и дочерние компоненты | 9 `Файл 01-structure.md`, подразделы `Граница компонента` и `Родительские и дочерние компоненты`; 14 `Связи между файлами` | Нужны, чтобы агент не принял соседний компонент за часть текущего и не потерял дочерний комплект. |
| Термины, базовые состояния, роль-зависимые или динамические списки | 5 `Соответствие прототипу и переносимость`; 9 `Файл 01-structure.md`, подраздел `Термины и базовые состояния`; 11 `Файл 03-data-states-rules.md`, если термин влияет на правила состояния | Нужны для точной интерпретации терминов вроде `черновик`, `базовое отображение`, `текущий набор`. |
| Функциональные иконки и визуальные обозначения | 6 `Что фиксировать в компоненте`, подраздел `Функциональные иконки и визуальные обозначения`; 10 `Файл 02-behavior.md`, если иконка запускает действие | Нужны, чтобы агент описывал не просто знак, а функциональный элемент интерфейса. |
| Поведение пользователя и реакции интерфейса | 10 `Файл 02-behavior.md`; 12 `Таблицы и сценарии`; 14 `Связи между файлами`; 15 `Проверка перед публикацией` | Помогает отделить пользовательский путь от структуры и правил состояния. |
| Данные, состояния, счетчики, правила видимости, владельцы состояния | 6 `Что фиксировать в компоненте`, подраздел `Контракты взаимодействия компонентов`; 11 `Файл 03-data-states-rules.md`; 14 `Связи между файлами`; 15 `Проверка перед публикацией` | Нужны для описания источников данных, получателей результата и правил, действующих вне одного клика. |
| Сквозной пользовательский путь | 13 `Сквозные пользовательские пути`; 14 `Связи между файлами`; 15 `Проверка перед публикацией` | Нужны, чтобы связать действия нескольких компонентов без дублирования их спецификаций. |
| Реализация функциональности по документации | 2 `Главный принцип`; 5 `Соответствие прототипу и переносимость`; 6 `Что фиксировать в компоненте`; 10 `Файл 02-behavior.md`; 11 `Файл 03-data-states-rules.md`; 14 `Связи между файлами`; 15 `Проверка перед публикацией` | Нужны, чтобы реализация не оторвалась от прототипа, данных и контрактов компонентов. |
| Проверка качества после работы агента или субагента | Разделы, соответствующие измененным файлам, и 15 `Проверка перед публикацией` | QA-субагент проверяет не весь стандарт, а применимые правила плюс общий чеклист. |
| Изменение самого стандарта документации | Весь `docs/documentation-standards.md` | Стандарт можно править только с учетом полной иерархии правил. |

Если тип задачи не определен, главный агент читает заголовки `docs/documentation-standards.md`, выбирает ближайшие применимые разделы и фиксирует это в задаче для субагента. Субагенту передаются только выбранные разделы стандарта или краткая ссылка на них, а не весь файл.

## 5. Контекст для компонентной задачи

Для задачи по одному компоненту основной пакет состоит из файлов:

| Тип задачи | Документы |
|---|---|
| Верстка, состав, расположение, границы | `01-structure.md` нужного компонента. |
| Действия пользователя и реакции интерфейса | `01-structure.md`, `02-behavior.md`. |
| Состояния, счетчики, правила, владельцы состояния, контракты | `01-structure.md`, `02-behavior.md`, `03-data-states-rules.md`. |
| Реализация функциональности | Комплект `01/02/03`, `docs/component-code-map.md`, связанные файлы `src`. |

Если у компонента есть `README.md`, агент читает его перед файлами `01/02/03`.

Для задачи по компоненту агент сначала выбирает самый узкий тип задачи из таблицы. Полный комплект `01/02/03` выбирается только тогда, когда задача требует структуры, поведения и правил состояния одновременно.

## 6. Контекст для субагента

Субагент получает только пакет своей зоны ответственности.

Для аудита, создания и исправления компонентной документации используется профиль `docs/subagents/component-documentation-auditor.md`. Профиль задает роль субагента, ограничения, входной пакет и формат результата. Skill `component-spec-audit` задает рабочий порядок проверки.

| Роль субагента | Передаваемый контекст |
|---|---|
| Субагент аудита компонентной документации | `docs/subagents/component-documentation-auditor.md`, применимые разделы `docs/documentation-standards.md`, skill `component-spec-audit`, файлы целевого компонента по типу задачи. |
| Субагент основного компонента | `README.md` компонента, если есть; `01-structure.md`; `02-behavior.md`; `03-data-states-rules.md`. |
| Субагент дочернего компонента | Комплект дочерней папки и родительский `01-structure.md`, где описана связь с родителем. |
| Субагент соседнего компонента | Комплект своего компонента и короткая ссылка на компонент-источник связи. |
| Субагент реализации | Комплект компонента, `docs/component-code-map.md`, только релевантные файлы `src`. |
| Субагент проверки качества | Комплект измененного компонента, связанные user-flow и применимые разделы `docs/documentation-standards.md` по разделу 4 этого маршрута. |

Субагент не должен получать документы другого компонента только потому, что они находятся рядом в папке `docs/components`.

Субагент не сканирует проект целиком. Если переданный пакет не отвечает на вопрос, субагент возвращает запрос на конкретный недостающий документ и объясняет, какой факт нужно проверить.

Главный агент решает, расширять пакет или нет. Субагент не добавляет новые области контекста самостоятельно.

## 7. Когда добавлять связанные документы

Связанный документ добавляется в пакет контекста только при явной причине.

| Причина | Что добавить |
|---|---|
| В `01-structure.md` есть ссылка на дочерний компонент | Комплект дочернего компонента. |
| В границе компонента указан соседний компонент | `01-structure.md` соседнего компонента; `02` и `03` добавлять только при изменении поведения или состояния. |
| Действие одного компонента меняет состояние другого | `03-data-states-rules.md` обоих компонентов. |
| Сценарий проходит через несколько компонентов | Соответствующий файл из `docs/user-flows/`. |
| Задача затрагивает данные или правила отбора | `03-data-states-rules.md`, `docs/entities/` при необходимости, файлы из `src/data` и `src/domain` по `component-code-map.md`. |
| Задача затрагивает реализацию интерфейса | `docs/component-code-map.md` и релевантные файлы `src/features`, `src/ui`, `src/styles`. |

Каждое расширение пакета фиксируется по схеме:

| Что указать | Пример |
|---|---|
| Недостающий факт | Неясно, какой компонент получает результат применения фильтров. |
| Добавляемый документ | `docs/components/tasks/03-data-states-rules.md`. |
| Причина добавления | Нужен контракт входящего состояния списка задач. |
| Граница чтения | Читать только правила входящих связей и отбора задач. |

## 8. Контроль перед передачей задачи субагенту

Перед запуском субагента главный агент проверяет:

- выбран конкретный компонент или дочерняя зона;
- указан ожидаемый результат работы;
- пакет контекста не содержит всю документацию целиком;
- указана запрещенная область, которую субагент не должен читать;
- указаны применимые разделы `docs/documentation-standards.md`, если задача связана с документацией или проверкой качества;
- в пакете есть `03-data-states-rules.md`, если задача касается функциональности;
- в пакете есть user-flow, если задача затрагивает сквозной пользовательский путь;
- в пакете есть `component-code-map.md`, если задача требует изменения кода.

## 9. Запрещенные способы передачи контекста

| Нельзя | Почему |
|---|---|
| Передать всю папку `docs/` | Агент может смешать правила, примеры и соседние компоненты. |
| Передать всю папку `docs/components/` | Субагент может принять соседний компонент за часть своей зоны. |
| Передать весь `docs/documentation-standards.md` для узкой компонентной задачи | Субагенту нужны применимые разделы стандарта, а не весь набор правил для всех случаев. |
| Передать только `01-structure.md` для функциональной задачи | Структура не содержит всех правил состояния и контрактов. |
| Передать только user-flow | User-flow не заменяет компонентные спецификации. |
| Передать только код без спецификации | Агент потеряет продуктовые границы и правила поведения. |
| Передать документы без указания цели чтения | Агент может использовать документ как общий фон и усреднить результат. |
| Передать соседний компонент без конкретного контракта связи | Агент может смешать границы текущего и соседнего компонента. |

## 10. Правило для будущей миграции на React

Слой корпоративной React-библиотеки, Storybook и дизайн-системы не входит в текущий маршрут.

После миграции нужно добавить отдельный слой контекста: продуктовый компонент, корпоративный компонент, ссылка на Storybook, правила применения и ограничения. До этого момента агент использует только продуктовую спецификацию, пользовательские пути и карту связи с текущим кодом.
