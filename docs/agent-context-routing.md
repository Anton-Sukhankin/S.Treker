# Маршрут контекста для AI-агента и субагентов

## 1. Назначение

Этот документ задает правило выбора ограниченного пакета контекста для работы с документацией, интерфейсом и функциональностью S-Tracker.

Цель маршрута: агент должен читать только те документы, которые нужны для конкретной задачи, не смешивать границы компонентов и не угадывать поведение по соседним файлам.

## 2. Базовый контекст главного агента

Главный агент перед планированием или реализацией читает:

| Документ | Зачем нужен |
|---|---|
| `AGENTS.md` | Постоянные правила проекта и ограничения на изменения. |
| `PROJECT-OVERVIEW.md` | Общее понимание продукта и назначения прототипа. |
| `docs/README.md` | Верхняя карта документации. |
| `docs/documentation-standards.md` | Обязательные правила оформления и проверки документации; для рабочей задачи читать только разделы, указанные в разделе 3 этого маршрута. |
| `docs/components/README.md` | Карта компонентных спецификаций. |

Главный агент не передает субагенту всю папку `docs/` или всю папку `docs/components/`.

## 3. Маршрутизация разделов `Documentation Standards`

`docs/documentation-standards.md` является основным стандартом, но для обычной компонентной задачи агент не должен читать его целиком. Главный агент сначала определяет тип задачи, затем выбирает только нужные разделы стандарта.

| Тип задачи | Какие разделы `docs/documentation-standards.md` читать | Зачем нужны |
|---|---|---|
| Любая новая или изменяемая компонентная спецификация | 1 `Назначение`; 2 `Главный принцип`; 3 `Правило максимальной конкретики`; 5 `Соответствие прототипу и переносимость`; 7 `Состав комплекта`; 8 `Последовательность чтения`; 14 `Проверка перед публикацией` | Задает общий уровень конкретики, состав комплекта и финальную проверку. |
| Структура, состав зон, расположение, границы компонента | 3 `Правило максимальной конкретики`; 6 `Что фиксировать в компоненте`; 9 `Файл 01-structure.md`; 13 `Связи между файлами`; 14 `Проверка перед публикацией` | Помогает описать видимую структуру, не смешивать компонент с контекстом и проставить ссылки. |
| Граница компонента, соседние области, родительские и дочерние компоненты | 9 `Файл 01-structure.md`, подразделы `Граница компонента` и `Родительские и дочерние компоненты`; 13 `Связи между файлами` | Нужны, чтобы агент не принял соседний компонент за часть текущего и не потерял дочерний комплект. |
| Термины, базовые состояния, роль-зависимые или динамические списки | 5 `Соответствие прототипу и переносимость`; 9 `Файл 01-structure.md`, подраздел `Термины и базовые состояния`; 11 `Файл 03-data-states-rules.md`, если термин влияет на правила состояния | Нужны для точной интерпретации терминов вроде `черновик`, `базовое отображение`, `текущий набор`. |
| Функциональные иконки и визуальные обозначения | 6 `Что фиксировать в компоненте`, подраздел `Функциональные иконки и визуальные обозначения`; 10 `Файл 02-behavior.md`, если иконка запускает действие | Нужны, чтобы агент описывал не просто знак, а функциональный элемент интерфейса. |
| Поведение пользователя и реакции интерфейса | 10 `Файл 02-behavior.md`; 12 `Таблицы и сценарии`; 13 `Связи между файлами`; 14 `Проверка перед публикацией` | Помогает отделить пользовательский путь от структуры и правил состояния. |
| Данные, состояния, счетчики, правила видимости, владельцы состояния | 6 `Что фиксировать в компоненте`, подраздел `Контракты взаимодействия компонентов`; 11 `Файл 03-data-states-rules.md`; 13 `Связи между файлами`; 14 `Проверка перед публикацией` | Нужны для описания источников данных, получателей результата и правил, действующих вне одного клика. |
| Сквозной пользовательский путь | 10 `Файл 02-behavior.md`; 12 `Таблицы и сценарии`; 13 `Связи между файлами` | Нужны, чтобы связать действия нескольких компонентов без дублирования их спецификаций. |
| Реализация функциональности по документации | 2 `Главный принцип`; 5 `Соответствие прототипу и переносимость`; 6 `Что фиксировать в компоненте`; 10 `Файл 02-behavior.md`; 11 `Файл 03-data-states-rules.md`; 13 `Связи между файлами`; 14 `Проверка перед публикацией` | Нужны, чтобы реализация не оторвалась от прототипа, данных и контрактов компонентов. |
| Проверка качества после работы агента или субагента | Разделы, соответствующие измененным файлам, и 14 `Проверка перед публикацией` | QA-субагент проверяет не весь стандарт, а применимые правила плюс общий чеклист. |
| Изменение самого стандарта документации | Весь `docs/documentation-standards.md` | Стандарт можно править только с учетом полной иерархии правил. |

Если тип задачи не определен, главный агент читает заголовки `docs/documentation-standards.md`, выбирает ближайшие применимые разделы и фиксирует это в задаче для субагента. Субагенту передаются только выбранные разделы стандарта или краткая ссылка на них, а не весь файл.

## 4. Контекст для компонентной задачи

Для задачи по одному компоненту основной пакет состоит из файлов:

| Тип задачи | Документы |
|---|---|
| Верстка, состав, расположение, границы | `01-structure.md` нужного компонента. |
| Действия пользователя и реакции интерфейса | `01-structure.md`, `02-behavior.md`. |
| Состояния, счетчики, правила, владельцы состояния, контракты | `01-structure.md`, `02-behavior.md`, `03-data-states-rules.md`. |
| Реализация функциональности | Комплект `01/02/03`, `docs/component-code-map.md`, связанные файлы `src`. |

Если у компонента есть `README.md`, агент читает его перед файлами `01/02/03`.

## 5. Контекст для субагента

Субагент получает только пакет своей зоны ответственности.

| Роль субагента | Передаваемый контекст |
|---|---|
| Субагент основного компонента | `README.md` компонента, если есть; `01-structure.md`; `02-behavior.md`; `03-data-states-rules.md`. |
| Субагент дочернего компонента | Комплект дочерней папки и родительский `01-structure.md`, где описана связь с родителем. |
| Субагент соседнего компонента | Комплект своего компонента и короткая ссылка на компонент-источник связи. |
| Субагент реализации | Комплект компонента, `docs/component-code-map.md`, только релевантные файлы `src`. |
| Субагент проверки качества | Комплект измененного компонента, связанные user-flow и применимые разделы `docs/documentation-standards.md` по разделу 3 этого маршрута. |

Субагент не должен получать документы другого компонента только потому, что они находятся рядом в папке `docs/components`.

## 6. Когда добавлять связанные документы

Связанный документ добавляется в пакет контекста только при явной причине.

| Причина | Что добавить |
|---|---|
| В `01-structure.md` есть ссылка на дочерний компонент | Комплект дочернего компонента. |
| В границе компонента указан соседний компонент | `01-structure.md` соседнего компонента; `02` и `03` добавлять только при изменении поведения или состояния. |
| Действие одного компонента меняет состояние другого | `03-data-states-rules.md` обоих компонентов. |
| Сценарий проходит через несколько компонентов | Соответствующий файл из `docs/user-flows/`. |
| Задача затрагивает данные или правила отбора | `03-data-states-rules.md`, `docs/entities/` при необходимости, файлы из `src/data` и `src/domain` по `component-code-map.md`. |
| Задача затрагивает реализацию интерфейса | `docs/component-code-map.md` и релевантные файлы `src/features`, `src/ui`, `src/styles`. |

## 7. Контроль перед передачей задачи субагенту

Перед запуском субагента главный агент проверяет:

- выбран конкретный компонент или дочерняя зона;
- указан ожидаемый результат работы;
- пакет контекста не содержит всю документацию целиком;
- указаны применимые разделы `docs/documentation-standards.md`, если задача связана с документацией или проверкой качества;
- в пакете есть `03-data-states-rules.md`, если задача касается функциональности;
- в пакете есть user-flow, если задача затрагивает сквозной пользовательский путь;
- в пакете есть `component-code-map.md`, если задача требует изменения кода.

## 8. Запрещенные способы передачи контекста

| Нельзя | Почему |
|---|---|
| Передать всю папку `docs/` | Агент может смешать правила, примеры и соседние компоненты. |
| Передать всю папку `docs/components/` | Субагент может принять соседний компонент за часть своей зоны. |
| Передать весь `docs/documentation-standards.md` для узкой компонентной задачи | Субагенту нужны применимые разделы стандарта, а не весь набор правил для всех случаев. |
| Передать только `01-structure.md` для функциональной задачи | Структура не содержит всех правил состояния и контрактов. |
| Передать только user-flow | User-flow не заменяет компонентные спецификации. |
| Передать только код без спецификации | Агент потеряет продуктовые границы и правила поведения. |

## 9. Правило для будущей миграции на React

Слой корпоративной React-библиотеки, Storybook и дизайн-системы не входит в текущий маршрут.

После миграции нужно добавить отдельный слой контекста: продуктовый компонент, корпоративный компонент, ссылка на Storybook, правила применения и ограничения. До этого момента агент использует только продуктовую спецификацию, пользовательские пути и карту связи с текущим кодом.
