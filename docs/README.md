# Документация S-Tracker

Документация нужна как спецификация интерфейса и поведения прототипа.

Главная цель: по документам должно быть понятно, что именно нужно собрать или изменить в интерфейсе.

## Главный источник

Основной источник по интерфейсу - `docs/components/`.

Документы компонентов должны быть конкретными чертежами: структура блока, состав элементов, действия пользователя, состояния и влияние на соседние области.

## Структура

- `components/` - спецификации интерфейсных компонентов.
- `entities/` - описание сущностей данных.
- `component-code-map.md` - связь спецификаций с файлами кода.
- `documentation-standards.md` - правила подготовки документации.
- `glossary.md` - словарь терминов проекта.
- `project-structure.md` - общая структура проекта.
- `user-flows/` - дополнительные сценарии, если они нужны отдельно от спецификации компонента.

## Действующие спецификации компонентов

Сейчас в новом формате описаны:

- `components/filters/` - фильтрация;
- `components/queues/` - очереди над списком задач.

Каждый из этих компонентов описан в трех файлах:

- `01-structure.md`;
- `02-behavior.md`;
- `03-data-states-rules.md`.

## Как давать контекст AI-агенту

Не передавать всю папку `docs`.

Для задачи выбрать только:

1. файл структуры нужного компонента;
2. файл поведения или правил, если задача касается логики;
3. точный запрос на изменение.
