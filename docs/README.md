# Документация S-Tracker

Документация в `docs/` описывает текущий рабочий прототип как спецификацию интерфейса, поведения и правил состояния.

Главная цель: по документам должно быть понятно, что именно нужно собрать, изменить, проверить или сохранить в интерфейсе без чтения всего проекта.

## Главный источник

Основной источник по интерфейсу - `docs/components/`.

Документы компонентов являются конкретными чертежами: структура блока, состав элементов, действия пользователя, состояния, связи с соседними областями и правила, которые влияют на функциональность.

Правила подготовки документации находятся в `documentation-standards.md`.

Маршрут чтения для главного агента и субагентов находится в `agent-context-routing.md`.

## Структура

- `components/` - спецификации интерфейсных компонентов.
- `component-code-map.md` - связь спецификаций с файлами реализации.
- `documentation-standards.md` - обязательный стандарт для всех документов в `docs/`.
- `documentation-standards-audit.md` - последний чекпоинт соответствия документации стандарту.
- `agent-context-routing.md` - правила выбора ограниченного пакета контекста для агента и субагентов, включая выбор применимых разделов `documentation-standards.md` по типу задачи.
- `entities/` - минимальный словарь общих интерфейсных данных прототипа.
- `glossary.md` - словарь терминов проекта.
- `project-structure.md` - общая структура проекта.
- `subagents/` - проектные профили временных субагентов для повторяемых задач с ограниченным контекстом.
- `templates/` - шаблоны создания и аудита документации.
- `user-flows/` - сквозные пользовательские пути, если сценарий выходит за рамки одного компонента.

## Действующие спецификации компонентов

Актуальная карта компонентов находится в `components/README.md`.

В новом формате описаны:

- `components/filters/` - панель фильтрации;
- `components/toolbar/` - панель инструментов списка задач;
- `components/left-navigation/` - левая навигация;
- `components/queues/` - этапы согласования над списком задач;
- `components/tasks/` - карточный и табличный список задач;
- `components/column-settings/` - настройки колонок таблицы;
- `components/bulk-actions/` - массовые действия над выбранными задачами;
- `components/pagination/` - пагинация списка задач.

`column-settings` является родительским компонентным комплексом. Внутри него отдельно описаны:

- `components/column-settings/presets/` - пресеты колонок;
- `components/column-settings/attribute-library/` - библиотека атрибутов.

Крупный компонент описывается комплектом:

- `01-structure.md`;
- `02-behavior.md`;
- `03-data-states-rules.md`.

Если компонент имеет дочерние зоны, карта комплекта может быть вынесена в `README.md` внутри папки компонента.

## Как давать контекст AI-агенту

Не передавать всю папку `docs` или `docs/components`.

Для конкретной задачи выбирать только:

1. обязательный проектный контекст;
2. карту компонентов;
3. комплект нужного компонента;
4. связанные дочерние или соседние компоненты по ссылкам;
5. user-flow, если задача затрагивает сквозной пользовательский путь;
6. `component-code-map.md` и файлы `src`, если задача включает реализацию функциональности.

Подробное правило выбора контекста описано в `agent-context-routing.md`. В нем же указано, какие разделы `documentation-standards.md` нужны для структуры, поведения, состояний, пользовательских путей, реализации функциональности и проверки качества.

## Служебные документы для создания и проверки спецификаций

| Документ | Роль |
|---|---|
| `documentation-standards.md` | Нормативный стандарт. Имеет приоритет над шаблонами, профилями субагентов и skill. |
| `templates/component-template.md` | Практический каркас для создания или аудита комплекта `01/02/03`. |
| `subagents/component-documentation-auditor.md` | Профиль временного субагента для проверки, исправления и создания компонентной документации. |
| `../skills/component-spec-audit/SKILL.md` | Рабочая процедура аудита компонентных спецификаций. |
| `documentation-standards-audit.md` | Чекпоинт последней проверки соответствия документации стандарту. Не является инструкцией. |

Эти документы не должны дублировать друг друга полностью. Стандарт задает правила, шаблон дает форму, профиль задает роль субагента, skill задает порядок работы, audit фиксирует результат проверки.
